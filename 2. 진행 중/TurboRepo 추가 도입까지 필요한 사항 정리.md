### Npm WorkSpace 를 사용한 모노레포 구성 완료 후 TurboRepo 추가 환경 구성


##### 터보레포는 모노레포를 만드는 도구가 아니라, 이미 구성된 모노레포 구조를 빠르고 일관성 있게 운영할 수 있도록 만들어주는 실행 엔진
##### 빌드/테스트/린트/타입체크를 더 빠르고 예측 가능하며, 적은 낭비로 돌리기 위해 도입 필요


#### 1. TurboRepo 도입 필요성
- 현재는 Npm WorkSpace 를 사용해서 모노레포 방식의 폴더 구조와 공통 프로젝트 사용 방식 적용 모두 완료한 상태
- 그럼에도 TurboRepo 방식은 도입이 필요할 것으로 생각됨
- 이유는
	1. 캐싱 ( 로컬 / CI )
		- 변경 없는 패키지는 build skip
		- CI 빌드 시간 감소
	
	2. 의존성 기반 실행 순서 자동화
		- 앱이 내부 패키지에 의존하면 dependensOn: ["^build"] 같은 설정으로 공통 패키지를 먼저 빌드한 뒤 앱을 빌드하게 만들 수 있다.
		- 공식 문서에서도 packages.json 의존관계를 사용해 패키지 관계를 이해하고, task 를 순서에 맞게 실행한다고 설명한다.
	
	3. 병렬 실행 최적화
		- 터보레포는 task graph를 바탕으로 "동시에 돌려도 되는 작업"을 병렬화하고 "선행 조건이 필요한 작업"은 순서를 지켜 실행한다.
		- task graph와 scheduler를 통해 incremental concurrent task running을 수행한다. 즉, 단순히 여러 명령을 한번에 날리는 수준이 아니라, 그래프 기반으로 병렬성과 안정성을 함께 가져간다.
	
	4. 변경 영향 범위만 실행 가능
		- 전체 빌드 대신 변경된 패키지와 그 영향권만 빌드를 실행하는 방식으로 CI, 로컬 빌드 시간이 줄어든다.
	
	5. CI 최적화
		- 로컬 캐시 뿐 아니라 remote 캐시를 지원해서 하나의 머신이나 하나의 CI job 에서 만든 결과를 다른 환경이 재사용 할 수 있다.
		- 중복 계산 자체를 줄이는 방식
	
	6. 개발자 경험 표준화
		- 루트 turbo.json의 task 아래에서 build, test, lint, dev 같은 task를 공통 모델로 정의하면, 각 패키지는 같은 이름의 script만 가지면 된다.
		- task의 각 key가 워크스페이스 내 package.json 스크립트 이름과 매칭 된다. 이는 팀 차원에서 해당 저장소의 build 방식을 한 곳에서 설명할 수 있다는 뜻이다.
