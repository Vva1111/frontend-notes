- 가상 DOM의 주요 장점은 개발자가 직접 DOM 조작을 렌더러에 맡기고 선언적 방식으로 원하는 UI 구조를 프로그래밍 방식으로 생성, 검사 및 구성할 수 있는 기능을 제공한다는 것이다.

  

## Render Pipeline

- 아래 항목은 Vue 컴포넌트가 마운트 될 때 발생한다.
    
    1. **Compile** : Vue Template은 렌더 함수로 컴파일 된다.
        
        가상 DOM 트리를 반환하는 함수. 이 단계는 빌드 과정을 통해 미리 실행하거나 런타임 컴파일러를 사용해 즉석에서 실행할 수 있다.
        
    
    1. **mount** : 런타임 렌더러는 렌더 함수를 호출하고, 반환된 가상 DOM 트리를 탐색하고, 이를 기반으로 실제 DOM 노드를 생성한다. 이 단계는 ‘[반응 이펙트](https://ko.vuejs.org/guide/extras/reactivity-in-depth.html)’가 실행되므로 사용된 모든 반응형 의존성을 추적한다.
    
    1. **Patch** : 마운트 중에 사용된 반응형 의존성이 변경되면 이펙트가 다시 실행된다. 이번에는 업데이트 된 새로운 가상 DOM 트리가 생성된다. 런타임 렌더러는 새 트리를 탐색하고 이전 트리와 비교하고 필요한 업데이트를 실제 DOM에 적용한다.

## Template / Render 함수

- Vue 템플릿은 가상 DOM 렌더 함수로 컴파일 된다.

- Vue는 템플릿 컴파일 단계를 건너뛰고 렌더 함수를 직접 작성할 수 있는 API를 제공한다.

- 렌더 함수는 JavaScript의 모든 함수를 사용해 vnode로 작업할 수 있기 때문에 매우 동적인 로직을 다룰 때 template보다 더 유연하다.

- Vue가 기본적으로 Template을 권장하는 이유는,
    
    1. Template 은 실제 HTML에 더 가깝다. 이를 통해 기존 HTML 스니펫을 재사용하고, 접근성 모범 사례를 적용하고, CSS로 스타일을 지정해 디자이너가 쉽게 이해하고 수정할 수 있다.
    
    1. Template 문법은 정적으로 분석하기가 더 쉽다. 이를 통해 Vue의 템플릿 컴파일러는 가상 DOM의 성능을 향상시키기 위해 많은 컴파일 시간 최적화를 적용할 수 있다,
    
    1. 렌더링 함수는 일반적으로 매우 동적인 렌더링 로직을 처리해야 하는 재사용 가능한 컴포넌트에서만 사용된다.
    
      
    

## **Compiler-Informed Virtual DOM**

- React와 대부분의 가상 DOM 방식은 조정 알고리즘에 할당되는 가상 DOM 트리에 대해 어떠한 가정도 할 수 없으므로, 정확성을 보장하기 위해 트리를 완전히 탐색하고 모든 vnode의 props를 비교해야 한다.
    
    또한, 트리의 일부가 변경되지 않더라도 다시 렌더링 할 때마다 항상 새로운 vnode가 생성되어 불필요한 메모리 부하가 발생한다.
    
    이는 가상 DOM의 가장 비판적 측면 중 하나이며, 다소 무차별적일 수 있는 조정 프로세스는 선언성과 정확성에 대한 대가로 효율성을 희생한다.
    
      
    

하지만 Vue에서 프레임워크는 컴파일러와 런타임을 모두 제어한다.

- 이를 통해 밀접하게 연결된 렌더러만 활용할 수 있는 많은 컴파일-타임 최적화를 구현할 수 있다.

- 컴파일러는 템플릿을 정적으로 분석하고, 생성된 코드에 힌트를 남겨 런타임이 가능할 때마다 바로 가기를 사용할 수 있도록 한다.

- 동시에 edge 케이스에서 보다 직접적인 제어를 위해 사용자가 렌더 함수 레이어로 드롭다운 할 수 있는 기능을 계속 유지한다. 이 방식을 Complier-Informed Virtual DOM(컴파일러에 알려진 가상 DOM)이라고 한다.

  

## Vue Template Compiler가 가상 DOM 런타임 성능 향상을 위해 수행하는 주요 최적화

  

### 1. **정적 호이스팅**

- Template 에는 동적 바인딩이 포함되지 않은 부분이 있는 경우가 많다.
    
    ```jsx
    <div>
      <div>foo</div> <!-- 호이스트(hoist) 됨 -->
      <div>bar</div> <!-- 호이스트 됨 -->
      <div>{{ dynamic }}</div>
    </div>
    ```
    

- 위 코드에서 foo, bar div는 정적이므로 리렌더에서 vnode를 다시 생성하고 이를 비교하는 것은 불필요하다.

- Vue 컴파일러는 vnode 생성 함수 호출 시, 자동으로 호이스트 해 모든 렌더에서 동일한 vnode를 재사용한다. 렌더는 이전 vnode와 새 vnode가 동일함을 발견하면 비교를 건너 뛸 수 있다.

- 또한, 정적 엘리먼트가 충분히 연속적으로 존재하면, 이러한 모든 노드에 대한 일반 HTML 문자열을 포함하는 단일 ‘정적 vnode’로 압축된다.

- 이러한 ‘정적 vnode’는 `innerHTML` 을 직접 설정해 마운트된다.

- 또한, 초기 마운트 시, 해당 DOM 노드를 캐시한다.

- 동일한 컨텐츠가 앱의 다른 곳에서 재사용 되는 경우, 기본 `cloneNode()` 를 사용해 새 DOM 노드가 생성되며, 이는 매우 효율적이다.

  

### 2. 패치 플래그

- 동적 바인딩이 있는 단일 엘리먼트의 경우, 컴파일 시 많은 정보를 추론할 수도 있다.

- 이러한 엘리먼트에 대한 렌더 함수 코드를 생성할 때, Vue는 vnode 생성 호출에서 직접 각 엘리먼트에 필요한 업데이트 유형을 인코딩한다.
    
    ![[Untitled 364.png]]
    

- 위 코드에서 마지막 인자 2는 ‘패치 플래그’이다. 엘리먼트는 단일 숫자로 병합 될 여러 패치 플래그를 가질 수 있다. 그 후 런타임 렌더러는 비트 연산을 사용해 플래그에 대해 검사해 특정 작업의 수행 여부를 결정할 수 있다.

- 비트 단위 검사는 매우 빠르다. 패치 플래그를 사용하면 Vue는 동적 바인딩으로 엘리먼트를 업데이트 할 때 필요한 최소한의 작업을 수행할 수 있다.

- 또한, vnode가 가진 자식 유형을 인코딩한다. 예를 들어, 여러 루트 노드가 있는 쳄플릿은 fragment로 표시된다. 대부분의 경우 이러한 루트 노드의 순서가 절대 변경되지 않는다는 것을 알고 있으므로, 이 정보를 패치 플래그로 런타임에 제공할 수도 있다.

  

### 3. Tree Flattening

- 개념적으로 block은 내부 구조가 안정적인 템플릿의 일부이다. 이 경우 전체 템플릿은 `v-if` 및 `v-for`와 같은 구조적 디렉티브를 포함하지 않기 때문에 단일 블록을 가진다.

- 각 block은 패치 플래그가 있는 (직계 자식 뿐만 아니라)모든 하위 노드를 추적한다.
    
    ![[Untitled 365.png]]
    

- 그 결과, 동적 하위 노드만 포함하는 병합된(flattened) 배열이 생성된다.
    
    ![[Untitled 366.png]]
    

- 이 컴포넌트를 다시 렌더링해야 할 때, 전체 트리 대신 병합된 트리만 탐색하면 된다.
    
    이를 ‘트리 병합(tree flattening)’이라고 하며, 가상 DOM 조정 중에 통과해야 하는 노드의 수를 크게 줄인다.( 템플릿의 모든 정적인 부분은 효과적으로 건너뛴다. )
    

- `v-if` 및 `v-for` 디렉티브는 새 블록 노드를 생성한다.
    
    ![[Untitled 367.png]]
    

- 자식 블록은 부모 블록의 동적 자손 배열 내에서 추적된다. 이는 상위 블록에 대한 안정적인 구조를 유지하게 해준다.
