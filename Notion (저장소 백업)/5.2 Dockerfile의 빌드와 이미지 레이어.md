# Dockerfile로부터 Docker 이미지 만들기

`docker build` 명령을 통해 Dockerfile에 정의된 구성 정보를 바탕으로 Docker 이미지를 생성할 수 있다.

## `docker build` 명령의 서식

```bash
docker build -t <생성할 이미지명><:태그명> [Dockerfile의 위치]
```

- **옵션**
    
    - `-f <파일명> <파일 위치>`: Dockerfile이 아닌 별도의 파일명을 지정할 때 `[Dockerfile의 위치]` 대신 사용
    
    - `-no-cache`: 중간 이미지를 재사용하고 싶지 않은 경우 지정
    

  

### ==예)== 베이스 이미지만 지정한 Dockerfile 빌드

**1️⃣ Dockerfile 생성**

```bash
$ mkdir sample && cd $-
$ touch Dockerfile

$ ls
Dockerfile
```

**2️⃣ Dockerfile 작성**

```docker
# 베이스 이미지 설정
FROM centos:centos7
```

**3️⃣ Dockerfile로부터 sample 1.0, sample 2.0 이미지 생성**

![[155p.Dockerfile로_이미지_생성.png]]

- `pwd`: 현재 디렉토리 경로 확인

- `docker build -t sample:1.0 /home/sunflower/sample`:  
    `/home/sunflower/sample` 경로의 Dockerfile로부터 버전이 `1.0`인 `sample` 이미지 생성
    
    - Dockerfile에 작성한 베이스 이미지를 Docker 레지스트리에서 다운로드(pull)받은 적이 없는 경우, 다운로드 받는 단계가 추가되어 시간이 더 걸림
    
    - Docker는 이미지 빌드 시 자동으로 중간 이미지(`centos:centos7`)를 생성하여 동일한 베이스 이미지를 바탕으로 다른 이미지를 빌드할 때 재이용함으로써 빠르게 빌드함
        
        - ==**(책 설명)**== `Using cache`라고 표시됨
        
        - ==**(Ubuntu 22.04 실행 결과)**== `CACHED`라고 표시됨
        
    

- `docker image ls`: 이미지 목록 확인

> [!important]
> 
> ==**책 설명과 실행 결과가 다른 부분 (Ubuntu 22.04)**==
> 
> > ==**(p.157)**== 여기서 확인해 두고 싶은 것은 베이스 이미지 centos:centos7과 작성한 두 개의 이미지의 이미지 ID가 모두 똑같다는 점입니다. 이것은 이미지로서는 각각 다른 이름이 붙어 있지만 그 실체는 모두 동일한 이미지라는 것을 나타냅니다.
> 
> 1. `**centos**` **이미지는 목록에 표시되지 않음**  
>     도커 빌드 시 Dockerfile에서는 베이스 이미지를 가져와서(final image를 빌드할 때 사용) 해당 레이어를 캐시로 활용한다. 따라서 Dockerfile에 작성된 내용은 최종 이미지(`sample:1.0`)만 빌드하게 되어, `centos:centos7` 이미지는 빌드된 결과물로 따로 생성되지 않는다.
> 
> 1. **이미지 ID가 다름**  
>     이미지 ID는 Dockerfile의 명령어뿐만 아니라 빌드 시 생성되는 메타데이터(예: 생성 시간 등)를 포함한 내용으로부터 생성된다. 따라서 동일한 Dockerfile을 사용하더라도 빌드할 때마다 생성되는 메타데이터가 달라져서 최종 이미지 ID도 달라진다.

  

### ==예)== Dockerfile 이외의 파일명을 지정하여 빌드

파일명이 Dockerfile이 아닌 경우 지정하여 빌드하는 방법이다.

```bash
docker build -t sample -f Dockerfile.base .
```

- `-f Dockerfile.base`: `-f` 옵션으로 `Dockerfile.base` 파일 지정

- `.`: 지금 위치해 있는 디렉토리를 가리킴

  

### ==예)== 표준 입력에서의 빌드

표준 입력을 경유하여 Dockerfile을 지정해 빌드하는 방법이다. 로컬 파일 시스템의 컨텍스트를 사용하지 않고, 표준 입력으로 전달된 Dockerfile 내용을 읽어 이미지를 빌드한다.

> [!important]
> 
> **표준 입력 (Standard Input)**
> 
> 터미널이나 콘솔에서 프로그램으로 입력을 주는 기본 통로  
> 
> > **로컬 컨텍스트 vs 표준 입력**
> 
> 로컬 컨텍스트를 사용하면 `COPY`, `ADD`와 같은 명령이 빌드 디렉토리 내 파일을 참조할 수 있지만 표준 입력을 사용하면 Docker를 빌드할 때 지정한 Dockerfile 내용만 전달된다.

```bash
docker build - < Dockerfile
```

- **옵션**
    
    - `-`: 표준 입력을 의미하는 옵션
    
    - `<`: 리다이렉션 옵션
    

- 위 명령을 사용하는 경우 빌드에 필요한 파일을 포함시킬 수 없으므로, 빌드에 필요한 파일이 있는 경우에는 Dockerfile과 빌드에 필요한 파일을 tar와 같은 압축 파일 안에 모아두고 표준 입력에서 지정함

  

### ==예)== 압축 아카이브에 의한 표준 입력에서의 빌드

이미지 안에 추가하고 싶은 dummyfile과 Dockerfile을 모아놓은 압축 파일로부터 이미지를 빌드하는 예이다.

```bash
docker build - < docker.tar.gz
```

> [!important]
> 
> **참고 명령**
> 
> - **tar.gz 압축파일 생성:** `tar -zcvf docker.tar.gz Dockerfile dummmyfile`
> 
> - **tar.gz 압축파일 내용 확인:** `tar tvfz docker.tar.gz`

# Docker 이미지 레이어 구조

Dockerfile을 빌드해 Docker 이미지 생성 시 Dockerfile의 명령마다 이미지를 생성하며, 레이어 구조이다.

![[Docker_이미지_레이어.png]]

- 공통의 베이스 이미지를 바탕으로 여러 개의 이미지를 생성한 경우, 베이스 이미지의 레이어가 공유됨
    
    - 이미지를 겹침으로써 디스크 용량을 효율적으로 이용함
    

- Docker Hub도 이미지를 레이어로 겹쳐서 공통되는 이미지를 공유하여 관리함
