( Representational State Transfer )

- 2000년 로이 필딩이 HTTP가 설계의 우수성에 비해 제대로 사용되어지지 못하다 생각해
    
    웹의 장점을 최대한 활용할 수 있도록 개발한 아키텍쳐 방식
    

- HTTP URI로 잘 표현된 리소스에 대한 행위를 HTTP Method로 정의한다

- 리소스는 json, xml, yaml 등의 다양한 표현 언어로 정의된다.
    
    즉, URI를 이용해 제어 할 자원을 명시하고, HTTP를 이용해서 제어 명령을 내린다.
    

  

- REST API의 구성
    
    1. 자원 ( RESOURCE ) : URI
    
    1. 행위 ( VERB ) : HTTP METHODS
    
    1. 표현 ( REPRESENTATIONS )
    

  

- REST 인터페이스의 원칙에 대한 가이드
    
    1. 자원의 식별
        
        - 요청 내에 기술된 자원을 식별할 수 있어야 한다.
        
        - 웹 기반에서는 URI를 예로 들 수 있다.
        
        - 자원 그 자체는 클라이언트가 받는 문서와는 개념적으로 분리되어있다.
        
    
    1. 메세지를 통한 리소스의 조작
        
        - 클라이언트가 어떤 자원을 지칭하는 메세지와 특정 메타데이터만 가지고 있다면
            
            서버 상의 해당 자원을 변경, 삭제할 수 있는 충분한 정보를 가지고 있는 것이다.
            
        
    
    1. 자기서술적 메세지
        
        - 각 메세지는 자신을 어떻게 처리해야 하는지에 대한 충분한 정보를 포함해야 한다.
        
        - 예로, MIME type과 같은 인터넷 미디어 타입을 전달한다면,
            
            그 메세지에는 어떤 파서를 이용해야 하는지에 대한 정보도 포함해야 한다.
            
            미디어 타입만 가지고도 클라이언트는 어떻게 그 내용을 처리해야 하는지
            
            알 수 있어야 한다는 뜻이다.
            
        
        - 메세지를 이해하기 위해 내용까지 살펴봐야 한다면, 해당 메세지는 자기서술적 메세지가 아니다.
        
        - 예를 들어, application/xml 이라는 미디어 타입은 실제 내용을 다운 받지 않으면
            
            메세지만 가지고는 무엇을 해야 할 지 충분히 알 수 없다
            
        
    

  

- REST API 의 특징
    
    1. uniform interface : URI로 지정한 리소스에 대한 조작을 통일되고 한정적인 인터페이스로
        
        수행하는 아키텍쳐 스타일
        
    
    1. stateless : 작업을 위한 상태 정보를 따로 저장하거나 관리하지 않는다
    
    1. cacheable : HTTP 웹 표준을 그대로 사용하기 때문에 기존 웹 인프라를 그대로 활용 가능
        
        > 캐싱 기능 적용 가능
        
    
    1. self-descriptiiveness ( 자체 표현 구조 ) : REST API 메세지만 보고도 쉽게 이해할 수 있는
        
        자체 표현 구조로 이루어져 있다.
        
    
    1. cilent - server 구조 : 클라이언트와 서버에서 개발해야 할 내용이 명확하고 의존성이 낮다
    
    1. 계층형 구조 : REST 서버는 다중 계층으로 이뤄져 있으며, 보안, 로드 밸런싱,
        
        암호화 계층을 추가해 구조 상의 유연성을 등 수 있다.
        
        [[PROXY]], [[GATEWAY]] 와 같은 네트워크 기반의 중간 매체를 사용할 수 있게 한다.
        
          
        
    

- REST API의 주요한 목표
    
    1. 구성 요소 상호작용의 규모 확장성
    
    1. 인터페이스의 범용성
    
    1. 구성 요소의 독립적인 배포
    
    1. 중간적 구성요소를 이용해 응답 지연 감소, 보안 강화, 레거시 시스템 인캡슐레이션
    

- REST API 디자인 가이드
    
    1. URI는 명사를 사용하며, method(get, post ~~)에 대한 표현이 들어가지 않는다
    
    1. URI 마지막에는 ‘/’ 를 사용하지 않는다
    
    1. ‘_’ 는 경로에 사용하지 않으며, 가독성을 위해서는 ‘-’ 를 사용한다
    
    1. URI 경로는 소문자를 사용한다
    
    1. 확장자는 경로에 포함하지 않는다.
