- Just-Time-In Complation 방식

![[Untitled 300.png]]

→ JavaScript JITC는 보통 위와 같은 방식으로 수행된다.

→ 현재 많이 쓰이고 있는 JavaScript 엔진은 모두 JITC 방식을 사용한다.

→ JavaScript는 기본적으로 text 형태로 배포되기 때문에 이를 수행하기 위해서는 변환이 필요하다.

1. source code를 parsing하여 중간 언어(Byte Code) 형태로 변환한다.

1. 인터프리터 모드의 경우, Byte Code를 하나씩 읽어가며 수행하고,

1. JIT 컴파일 모드라면 생성된 Byte Code를 기반으로 Native Code로 컴파일 한 후 수행한다.

1. JITC는 컴파일 과정 자체가 수행 중에 발생하기 때문에 overhead가 되기 때문에 컴파일에 많은 시간을 할애할 수 없다.

1. 그러므로, 전체 코드를 훑어서 최적화 하는 방식은 사용할 수 없고, 보통은 아주 최소한의 최적화만 적용해 Native Code를 생성하게 된다.

1. 그럼에도 인터프리터 수행 시간보다 Native Code의 수행 성능이 월등히 좋기 때문에 Java와 같은 언어에서는 JITC를 사용해 컴파일한다.

→ JavaScript에서는 변수의 타입이 동적으로 결정되고, class 대신 object로 상속되는 prototype-based 방식을 사용한다.

- JavaScript의 JITC는 모든 예외적인 케이스를 다 고려해 코드를 생성해야 하고, 단순한 덧셈 계산식의 경우에도 여러 예외 케이스를 다 고려해야 하기 때문에 많은 양의 Native Code가 생성된다.

![[Untitled 301.png]]

그림. 덧셈 계산식을 Native Code로 변환하는 과정

→ Slow Case: Native Code로 표현하면 양이 많아지는 동작들을 Naive Code로 뽑아내는 대신, 미리 엔진 내부에 C를 사용해서 구현된 helper function을 호출해 동작을 수행하는 경우를 말한다.

→ 수 많은 동적 케이스들을 Native Code로 변환하게 된다면 Native Code의 양이 워낙 많아지기 때문에 helper function을 사용하는데, 이러한 helper function은 인터프리터 모드로 사용할 때와 똑같은 코드를 사용한다.

→ 즉, JITC로 Native Code를 수행한다 해도, 많은 부분은 인터프리터와 별반 차이가 없거나, 혹은 다른 언어의 JITC보다 더 비효율적일수도 있다.

→ 또한, Web Page의 Layout을 조작하거나, 사용자 입력 반응 방식을 많이 구현하고 상대적으로 연산 작업이 적은 JavaScript의 특성 상, 반복 수행되는 구간이 (상대적으로)적기 때문에 Native Code를 수행하는 시간에 비해 그 Native Code를 만들어 내는 시간(Compile Overhead)이 상대적으로 커지게 된다.

→ 이러한 이유들로, 연산 반복 수행 구간이 적은 JavaScript의 경우, 인터프리터로 수행하는 것이 더 효율적이다.

→ 하지만 최근에는 JavaScript가 단순히 Event 처리 용도로만 사용되는 것이 아닌, 비즈니스 로직에도 어느정도 관여를 하게 되면서, JITC 방식을 완전히 버릴 수는 없기 때문에 최근의 JavaScript 엔진들은 대부분 Adaptive compliation 방식을 사용하고 있다.

- Adaptive JIT Compilation이란, 모든 코드를 일괄적으로 같은 수준의 최적화를 적용하는 것이 아닌, 반복 수행되는 정도에 따라 서로 다른 최적화 수준을 적용하는 방식이다.

- 기본적으로 모든 코드는 인터프리터 방식으로 수행되며, 반복적인 연산 작업이 발견되면, 해당 부분에 대해서만 JITC를 적용해 Native Code로 수행한다.

- 최근의 엔진들은 JITC도 여러 단계로 나누어 적용한다. 처음에는 최소한의 ‘Baseline-JITC’만 적용해 컴파일을 수행하다가, 연산 반복이 자주 수행되는 코드 지점에서 ‘Optimizing-JITC’로 컴파일을 수행해 효율적인 코드를 생성한다.

![[Untitled 302.png]]

그림. Baseline-JITC와 Optimizing-JITC 동작

- Crankshaft는 인터프리터가 없고 Baseline-JITC 부터 시작한다.

- Adaptive JITC를 사용하는 엔진들은 공통적으로 runtime profiler가 함수의 수행 빈도를 기록하고, 사용되는 변수의 타입이나 값을 기록해 두었다가 optimizing JITC를 적용할 때 이들 정보를 이용해 예전 JITC에서 생성했던 예외 처리 루틴을 대폭 생략한 효율적인 코드를 생성한다.

- 다만 이 작동방식은 ‘profiling을 수행하는 동안 특정 변수의 타입이 변하지 않았다면 그 이후에도 그 변수는 타입이 변하지 않을 가능성이 매우 높을 것이다’ 라는 가정을 바탕에 둔 최적화 방식이다. ( 중간에 타입이 변해서 예외 상황이 발생하는 경우 OPtimizing 방식이 아닌 Baseline 방식으로 수행하게 된다.)

  

### 1. 복잡한 연산 반복이 많지 않은 경우, 인터프리터가 효율이 좋다

### 2. 최근의 JavaScript에는 JITC가 효율적이다.

### 3. 두 가지 성향의 코드에 대한 성능을 모두 만족하기 위해 최근 JavaScript엔진들에는 Adaptive JITC 방식을 사용하고 있다.

### 4. Adaptive JITC는 type profiling을 수행하므로, 변수의 type이 변하지 않는다면 높은 성능을 얻을 수 있다.(loop 도중 type이 변하게 되면 큰 패널티가 발생할 수 있다.)

> 그림. JavaScript의 JIT 과정
> 
>   
> 
> - 자바스크립트 엔진은 자바스크립트 코드를 받은 뒤, Tokenizing을 통해 ‘토큰’이라는 단위로 잘게 쪼갠 후, 파싱 과정을 거쳐 의미 단위로 트리 구조 형태로 정리되는 AST(추상 구문 트리)형태를 가진다.
> 
> - 인터프리터에서는 AST를 가상 머신이 이해하고, 기계어로 변환 시킬 수 있도록 하는 Byte Code로 변환하는 일을 맡는다.
> 
> - 프로그램을 실제 실행하는 시점에 바이트 코드를 기계어로 번역하는 컴파일 기법
> 
> - 컴파일 언어의 경우, 연산 결과를 미리 기계어로 바꿔 놓는다는 특징이 있어 인터프리터보다 속도가 빠르다.
> 
> ```jsx
> function sum () {
> 	let result = 0
> 	for (let i = 1 ; i <= 10 ; i++){
> 		result += i;
> 	}
> 	return result;
> }
> sum() // for loop.. -> 55
> sum() // for loop.. -> 55
> sum() // for loop.. -> 55
> 
> // compile 결과
> sum() = 55
> sum() = 55
> sum() = 55
> ```
> 
> 예를 들어, 위와 같은 코드가 있다고 하면,
> 
> 인터프리터의 경우, 가장 첫 줄부터 실행해 나가는 방식이기 때문에 sum 함수를 만나고, 그 이후 for문을 거쳐야 결과가 도출되는 방식인데, 컴파일 과정은 sum 함수의 결과 값을 미리 기계어로 번역 해 놓는다는 특징이 있다.
> 
> - V8 엔진 역시 기존의 방식과 동일하게 파싱 이후 AST로 변환 된 내용을 인터프리터에 전달하는 과정에 새롭게 Profiler 라는 개념이 등장한다.
> 
> - Profiler는 인터프리터를 관찰하며 실행되는 코드를 계속해서 모니터링 하며, 코드 내에 반복 실행되는 것이 있다면 이를 컴파일러에게 넘겨 실시간으로 컴파일 하도록 한다. 이를 통해 최적화 된 바이트 코드를 생성한다.
> 
> - 위의 코드 예시에서 sum 함수와 같이 반복해서 진행될 여지가 있는 코드가 컴파일 대상이 된다.
> 
> - 이처럼 필요할 때만 컴파일을 진행하는 방식을 JIT(Just-In-Time) 컴파일러라고 부른다.
> 
> - Decompile : 컴파일러 진행 판단 착오일 경우, 되돌리는 과정

> [!important] JIT를 사용함으로써 얻을 수 있는 장점으로,
> 
> - 정적 컴파일처럼 미리 바이트 코드로 컴파일 하지 않고 실행 시간에 결정
> 
> - 정적 컴파일처럼 최적화를 일부 적용할 수 있다.

> [!important] V8 이전의 자바스크립트 가상 머신
> 
> - SpiderMonkey : mozilla firefox 브라우저의 자바스크립트 엔진. 최초로 자바스크립트 인터프리터를 구현한 엔진. 브렌던 아이크 개발, 넷스케이프 코퍼레이션에서 자바스크립트 언어를 만들 때 사용. 이후 JIT 컴파일러 추가
> 
> - JavaScriptCore : 애플의 브라우저 엔진인 Webkit에 사용되는 자바스크립트 엔진. 대표적으로 Safari 에서 작동한다. 초기에는 SpiderMonkey와 같이 인터프리터로 동작했지만 이후JIT 컴파일러가 포함되었다.
