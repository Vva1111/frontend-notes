## 호이스팅

- 변수의 선언문이 코드의 선두로 끌어올려진 것처럼 동작하는 자바스크립트 고유의 특징을 ‘호이스팅’이라고 한다.

- var, let, const, function, class 키워드를 사용해 선언한 모든 식별자는 호이스팅 된다.

  

### 변수 호이스팅

### var

![[Untitled 72.png]]

![[Untitled 73.png]]

  

- var 변수 선언은 선언과 동시에 암묵적으로 undefined 를 할당한다.

- var 키워드로 선언한 변수의 경우, 런타임 이전에 자바스크립트 엔진에 의해 암묵적으로 ‘선언 단계’와 ‘초기화 단계’가 한번에 진행된다.

![[Untitled 74.png]]

- 자바스크립트 코드는 인터프리터에 의해 한 줄씩 순차적으로 실행되므로 console.log(score); 가 실행되는 시점에는 score가 선언되지 않았기 때문에 참조 에러가 발생할 것 같지만, 실제로는 변수가 정상적으로 선언 되었고, console.log(score); 또한 정상적으로 작동한다.

- 이유는, 변수 선언이 소스 코드가 한 줄씩 순차적으로 실행되는 시점인 런타임에 실행되는 것이 아니라, 그 이전 단계에서 먼저 실행되기 때문이다.

- 자바스크립트 엔진은 소스 코드를 한 줄씩 순차적으로 실행하기에 앞서 소스 코드의 평가 과정을 거치는데, 이 때 자바스크립트 엔진은 변수 선언을 포함한 모든 선언문을 소스 코드에서 찾아내 먼저 실행한다.

- 따라서 변수 선언문 이전에 변수에 접근해도 스코프에 변수가 존재하기 때문에 에러가 발생하지 않고 초기화 된 갑인 undefined 값을 반환한다. 이후 변수 할당문에 도달하면 비로소 값이 할당되는데, 이 현상을 ‘호이스팅’이라고 한다.

- 즉, 자바스크립트 엔진은 변수가 어디에서 선언이 되었는지에 상관 없이 다른 코드보다 먼저 실행한다는 것이다. 그렇기 때문에 소스 코드의 어느 위치에서든 var 변수를 참조할 수 있는 것이다.

- 선언 단계에서 실행 컨텍스트의 *렉시컬 환경에 변수 식별자를 등록해 자바스크립트 엔진에 변수의 존재를 알린 후, 초기화 단계에서 undefined로 변수를 초기화한다. 따라서, 변수 선언문 이전에 변수에 접근해도 스코프에 변수가 존재하기 때문에 에러가 발생하지 않는다. 이후 변수 할당문에 도달하면 그때 값이 할당된다.

> [!important] 렉시컬 환경 : 코드가 어디서 실행되며, 주변에 어떤 코드가 있는지를 일컫는 말로, 코드의 문맥(Context)은 렉시컬 환경으로 이루어진다. 이를 구현한 것이 ‘실행 컨텍스트’ 이며, 모든 코드는 실행 컨텍스트에서 평가되고 실행된다.

  

- 자바스크립트는 var, let, const, class 함수 와 같은 모든 선언을 호이스팅 하지만 var, 함수 선언문을 제외한 let, const, class를 사용한 선언문은 호이스팅이 발생하지 않는 것처럼 동작한다.

  

### let

![[Untitled 75.png]]

![[Untitled 76.png]]

- 동일한 스코프 내에서 변수명 중복 선언 불가

- let 키워드로 선언한 변수의 경우, ‘선언 단계’와 ‘초기화 단계’가 분리되어 진행된다.

- 런타임 이전에 자바스크립트 엔진에 의해 암묵적으로 선언 단계가 먼저 실행되지만 초기화 단계는 변수 선언문에 도달했을 때 실행된다.

- 그렇기 때문에 초기화 단계 이전에 변수에 접근하려 하면 참조 에러가 발생한다.

- let 키워드로 선언한 변수는 스코프의 시작 지점부터 초기화 단계 시작 지점(변수 선언문) 까지 변수를 참조할 수 없다. 스코프의 시작 지점부터 초기화 시작 지점까지 변수를 참조할 수 없는 구간을 ‘일시적 사각지대’ 라고 한다.

![[Untitled 77.png]]

  

- 선언단계 : 변수를 실행 컨텍스트의 변수 객체(variable Object) 에 등록한다. 이 변수 객체는 스코프가 참조하는 대상이 된다.

- 초기화 단계 : 변수 객체에 등록된 변수를 위한 공간을 메모리에 확보한다. 이 단계에서 변수는 undefined로 초기화 된다.

- 할당 단계 : undefined로 초기화 된 변수에 실제 값을 할당한다.

![[Untitled 79.png]]

- let 키워드로 선언한 전역 변수는 var 키워드로 선언한 전역 변수와는 다르게 전역 객체 window의 프로퍼티가 아니다. 즉, window.foo와 같이 접근할 수 없다는 뜻이다. let 전역 변수는 보이지 않는 개념적인 블록(전역 렉시컬 환경의 선언적 환경 레코드) 내에 존재하게 된다.

### **const**

- 상수 선언을 위해 사용하는 키워드

- 호이스팅 동작 방식은 let과 동일하다.

- const 키워드로 변수 선언을 하기 위해서는 반드시 선언과 동시에 초기화를 해야 한다.

- const 키워드 변수는 재 할당이 금지되어 있다.

- const 키워드로 선언한 변수에 원시 값을 할당한 경우, 변수 값을 변경할 수 없다. 따라서, 쉽게 바뀌지 않는 값 등에 사용할 수 있다.

- 하지만 cosnt 키워드로 선언된 변수에 할당 된 객체 값은 변경할 수 있다.

### 함수 호이스팅

![[Untitled 83.png]]

![[Untitled 84.png]]

- 함수 선언문으로 정의한 함수는 선언문 이전에 호출할 수 있다.

- 함수 표현식으로 정의한 함수는 함수 표현식 이전에 호출할 수 없다.

- 이는 함수 선언문으로 정의한 함수와 함수 표현식으로 정의한 함수의 생성 시점이 다르기 때문이다.

- 함수 선언문 역시 변수 선언문과 동일하게 코드가 한 줄씩 순차적 실행되는 시점인 런타임 이전에 자바스크립트 엔진에 의해 먼저 실행된다. 다시 말해, 선언문으로 함수를 생성했을 때, 함수 객체는 런타임 이전에 먼저 생성이 되어 있는다는 의미다.

- 변수 호이스팅과의 차이점은, 호이스팅 된 변수는 암묵적으로 undefined로 초기화 되지만, 함수는 함수 객체로 초기화 된다는 점이다.

- 함수 표현식으로 호이스팅을 발생 시킬 시, 변수 선언은 런타임 이전에 undefined로 평가 되지만 변수 할당은 런타임에 되기 때문에 함수 표현식의 함수 리터럴도 런타임에 평가되어 함수 객체가 된다.

- 따라서, 함수 표현식을 사용해 함수를 정의하면 함수 호이스팅이 아니라 변수 호이스팅이 발생하게 된다.

- 함수 표현식 이전에 함수를 참조하면 undefined가 참조되며, 이 때 함수를 호출하면 undefined를 호출하는 것이므로 타입 에러가 발생한다.

- 따라서, 함수 표현식으로 정의한 함수는 반드시 함수 표현식 이후에 참조 또는 호출해야 한다.

- 함수 호이스팅은 함수를 호출하기 전에 반드시 함수를 선언해야 한다는 당연한 규칙을 무시하며, 이 같은 문제로 함수 선언식 보다는 표현식을 사용하는 것을 권장하고 있다.

  

## 스코프(유효 범위)

- 자바스크립트 뿐만 아니라 모든 프로그래밍 언어에서 중요한 개념

- 자바스크립트는 타 언어와 구별되는 특징이 있음

- 자바스크립트 엔진은 코드를 실행할 때 코드의 문맥을 고려한다. 코드가 어디서 실행되며, 주변에 어떤 코드가 있는지에 따라 같은 변수 이름이더라도 다른 결과를 낸다.

  

### 개념

- 모든 식별자 ( 변수 이름, 함수 이름, 클래스 이름 등) 는 자신이 선언 된 위치에 의해 자신이 유효한 범위, 즉 다른 코드가 변수 자신을 참조할 수 있는 범위가 결정된다. 즉, 스코프는 ‘식별자가 유효한 범위’ 를 말한다.

- 자바스크립트 엔진은 스코프를 통해 어떤 변수를 참조해야할 것인지를 결정한다. 따라서 스코프란, 자바스크립트 엔진이 식별자를 검색할 때 사용하는 규칙 이라고도 말할 수 있다.

- 스코프(유효 범위)를 통해 식별자인 변수 이름의 충돌을 방지해 같은 이름의 변수를 사용할 수 있게 한다. 스코프 내에서 식별자는 유일해야 하지만 다른 스코프에는 같은 이름의 식별자를 사용할 수 있다. 즉, 스코프는 네임 스페이스이다.

- let ,const 키워드로 선언된 변수 이름은 같은 스코프 내에서 중복이 허용되지 않는다.

  

### 전역 / 전역 스코프

- 전역 변수는 어디서든 참조할 수 있다

  

### 지역 / 지역 스코프

- 지역이란 함수 몸체 내부를 뜻한다.

- 지역 변수는 자신이 선언된 지역과 하위 지역(중첩 함수)에서만 참조할 수 있다. 즉, 지역 변수는 자신과 자신의 하위 지역 스코프에서만 유효하다.

  

## 스코프 체인

- 함수는 중첩될 수 있으므로, 함수의 지역 스코프도 중첩될 수 있다. 이는 스코프가 함수의 중첩에 의해 계층적인 구조를 갖는다는 것을 의미한다.

- 다시 말해서, 중첩 함수의 지역 스코프는 중첩 함수를 포함하는 외부 함수의 지경 스코프와 계층적 구조를 갖는다. 이 때, 외부 함수의 지역 스코프를 중첩 함수의 상위 스코프라고 한다.

![[Untitled 85.png]]

그림. 스코프 체인

- 이렇게 스코프가 계층적으로 연결된 것을 ‘스코프 체인’이라고 칭한다.

- 모든 스코프는 하나의 계층적 구조로 연결되며, 모든 지역의 최상위 스코프는 전역 스코프다.

  

### 스코프 체인의 변수 검색

![[Untitled 86.png]]

- 변수를 참조할 때, 자바스크립트 엔진은 스코프 체인을 통해 변수를 참조하는 코드의 스코프에서 시작해 상위 스코프 방향으로 이동하며 선언된 변수를 검색한다. 이를 통해 상위 스코프에서 선언한 변수를 하위 스코프에서도 참조할 수 있다.

- 자바스크립트 엔진은 절대 하위 스코프로 내려가면서 식별자를 검색하는 일은 없다.

- 상위 스코프에서 유효한 변수는 하위 스코프에서 자유롭게 참조할 수 있지만, 하위 스코프에서 유효한 변수를 상위 스코프에서는 참조할 수 없다는 것을 의미한다.

- 스코프 체인은 물리적인 실체로 존재한다.

- 자바스크립트 엔진은 코드(전역/ 함수)를 실행하기에 앞서 위 그림과 유사한 자료구조인 렉시컬 환경을 실제로 생성한다.

- 변수 선언이 실행되면 변수 식별자가 이 자료구조(렉시컬 환경)에 key로 등록되고, 변수 할당이 일어나면 이 자료구조의 변수 식별자에 해당하는 값을 변경한다. 또한, 변수의 검색도 이 자료구조 상에서 일어난다.

> [!important] 스코프 체인은 실행 컨텍스트의 렉시컬 환경을 단방향으로 연결 한 것이다. 전역 렉시컬 환경은 코드가 로드되면 곧바로 생성되고, 함수의 렉시컬 환경은 함수가 호출되면 곧바로 생성된다.

  

### 스코프 체인의 함수 검색

![[Untitled 88.png]]

  

- 함수 선언문으로 선언한 함수는 호이스팅이 발생해 런타임 이전에 함수 객체가 먼저 생성 된 이후에 자바스크립트 엔진은 함수 이름과 동일한 이름의 식별자를 암묵적으로 선언하고 생성된 함수 객체를 할당한다.

## 함수 레벨 스코프

- 지역 스코프는 코드 블록이 아닌 함수에 의해서만 생성된다.

- 자바스크립트의 var는 C 혹은 자바의 ‘블록 레벨 스코프( if, for, while 등 모든 코드 블록 내부가 지역 스코프를 만들 수 있는 개념’)’ 와는 다르게 ‘함수 레벨 스코프’로 동작한다.

- 이러한 자바스크립트 var 키워드의 경우, 함수 밖에서 선언 된 키워드는 코드 블럭(if, for 등) 내부에서 같은 식별자 이름으로 선언 되었어도 모두 전역 변수로 취급 받게 된다.

- 또한 위의 경우에 같은 식별자 이름으로 중복 변수가 선언 된 것이다.

- 이러한 경우, 의도치 않게 값이 변하게 되는 부작용이 발생할 수 있고, 변수의 의미가 명확해지지 않는다.

- let, const 키워드를 사용하면 블록 레벨 스코프를 지원하기 때문에 let, const 사용이 권장된다.

- 위의 실행 결과는 bar 함수의 상위 스코프가 무엇인지에 따라 결정된다.

- 함수를 어디서 호출했는지에 따라 함수의 상위 스코프가 결정 되는 것을 ‘동적 스코프’ 라고 하며, 이는 함수가 정의된 시점에는 해당 함수가 어디에서 호출될 지 알 수 없기 때문에 호출된 시점에 따라 동적으로 상위 스코프를 결정해야 한다.

- 함수를 어디서 정의했는지에 따라 함수의 상위 스코프를 결정하는 방식을 ‘렉시컬 혹은 정적 스코프’ 라고 한다. 동적 스코프 방식 처럼 상위 스코프가 동적으로 변하지 않고 함수 정의가 평가되는 시점에 상위 스코프가 정적으로 결정되기 때문이다.

- 자바스크립트를 포함한 대부분의 프로그래밍 언어가 렉시컬 스코프 방식을 사용한다.

- 함수가 호출되는 위치는 상위 스코프 결정에 어떠한 영향도 주지 않는다. 즉, 함수의 상위 스코프는 언제나 자신이 정의된 스코프다.

- 렉시컬 스코프는 함수 선언문, 함수 표현식에 상관 없이 정의된 위치에 따른르며, 상위 스코프를 기억한다. 함수가 실행 될 때마다 함수의 상위 스코프를 참조할 필요가 있기 때문이다.

- 그렇기 때문에 위의 예제에서 bar 함수는 정의 된 위치의 상위 스코프 변수 x=1을 출력하게 된다.
