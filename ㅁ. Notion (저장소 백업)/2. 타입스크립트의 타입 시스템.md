  

### Item 6. 편집기를 사용해 타입 시스템 탐색하기

- 타입 스크립트를 설치하면, 다음 두 가지를 실행할 수 있다.
    
    - 타입 스크립트 컴파일러(tsc)
    
    - 단독으로 실행할 수 있는 타입 스크립트 서버(tsserver)
    

- 보통은 타입 스크립트 컴파일러를 실행하는 것이 주 목적이지만, 타입 스크립트 서버 또한 ‘언어 서비스’를 제공한다는 점에서 중요하다.

- ‘언어 서비스’는 코드 자동 완성, 명세 검사, 검색, 리팩터링이 포함된다.

- 보통 편집기를 통해 언어 서비스를 제공받는데, 타입 스크립트 서버에서 언어 서비스를 제공하도록 설정하는 것이 좋다.

- 편집기는 코드 빌드, 타입 시스템을 익히는데 많은 도움을 줄 수 있으며, 타입 스크립트가 언제 타입 추론을 수행할 수 있는지에 대한 개념을 잡는데 도움을 줄 수 있다.

  

  

### Item 7. 타입이 값들의 집합이라고 생각하기

- 코드가 실행되기 전, 즉 타입 스크립트가 오류를 체크하는 순간에는 ‘타입’을 가지고 있다. ‘타입’은 할당 가능한 값들의 집합 이라고도 할 수 있다.

- 가장 작은 집합은 아무것도 포함하지 않는 공집합이며, 타입 스크립트에서는 never 타입이다.

- 그 다음으로 작은 집합은 한 가지 값만 포함하는 타입인 유닛 혹은 리터럴 타입이다.

- 두 개 혹은 세 개로 묶으려면 유니온 타입을 사용한다. ( type AB = ‘A’ | ‘B’ ; )

- 다양한 타입 스크립트 오류에서 ‘할당 가능한’이라는 표현을 확인할 수 있는데, 이 문구는 집합의 관점에서 ‘~의 원소(값과 타입의 관계)’ 또는 ‘~의 부분 집합(두 타입의 관계)’ 를 의미한다.

- 집합의 관점에서, 타입 체커의 주요 역할은 하나의 집합이 다른 집합의 부분 집합인지 검사는 것 이라고 볼 수 있다.

- 범위가 무한대인 number, string 등과 같은 타입은 원소들을 일일이 추가해서 만든 걸로 생각할 수 있다.

- & 연산자는 두 타입의 교집합을 계산한다.
    
    - 공통적인(교집합) 타입을 가지지 않는 두 interface는 공통으로 가지는 속성이 없기 때문에 타입을 공집합(never)으로 예상하기 쉽지만, 연산자는 인터페이스의 속성이 아닌 값의 집합(타입의 범위)에 적용된다.
    
    - 인터섹션 타입의 값은 각 타입 내의 속성을 모두 포함하는 것이 일반적인 규칙이다.
    

- | 유니온 타입
    
    - 공통적인 타입을 가지지 않는 두 인터페이스의 |(유니온)은 공집합(never)이어야 한다.
    

- keyof (A&B) = (keyof A) | (keyof B)
    
    keyof (A|B) = (keyof A) & (keyof B)
    

- 타입이 집합이라는 관점에서 extends의 의미는 ‘~에 할당가능한’ 과 비슷하게, ‘~의 부분 집합’ 이라는 의미로 받아들일 수 있다.`

- extends 키워드는 데네릭 타입에서 한정자로도 쓰이며, 이 문맥에서는 ‘~의 부분 집합’을 의미하기도 한다.

- string을 상속한다는 의미를 집합의 관점으로 생각해보면 쉽게 이해할 수 있다.
    
    - string의 부분 집합 범위를 가지는 어떤 타입이 된다.
    
    - 이 타입은 string 리터럴 타입, string 리터럴 타입의 유니온, string 자신을 포함한다.
    

- ‘할당될 수 없습니다.’는 상속의 관점에서 ‘상속할 수 없다’ 로 바꿀 수 있고, 두 표현 모두 ‘~의 부분 집합’의 의미로 받아들여도 문제가 없다.

- 타입들이 엄격한 상속 관계가 아닐 때는 집합 스타일이 더욱 바람직하다.
    
    - string | number 와 string | Date 사이의 인터섹션은 공집합이 아니며(string 이다.), 서로의 부분 집합도 아니다.
        
        - string | number, string(인터섹션), string | Date 와 같이 나눠진다.
        
    

- 타입이 값의 집합이라는 건, 동일한 값의 집합을 가지는 두 타입은 같다는 의미이다.
    
    - 두 타입이 의미적으로 다르고 우연히 같은 범위를 가진다고 하더라도 같은 타입을 두 번 정의할 필요는 없다.
    

  

  

  

### Item 8. 타입 공간과 값 공간의 심벌 구분하기

- 심벌(Symbol)은 타입 공간이나 값 공간 중의 한 곳에 존재한다.

- 타입 스크립트 코드에서 타입과 값은 번갈아 나올 수 있으며, 타입 선언(:) 또는 단언문(as) 다음에 나오는 심벌은 타입이고, = 다음에 나오는 모든 것은 값이다.

- typeof
    
    - 타입의 관점에서, typeof는 값을 읽어서 타입스크립트 타입을 반환한다.
    
    - 값의 관점에서 typeof는 자바스크립트 런타임의 typeof 연산자가 된다. 값 공간의 typeof는 대상 심벌의 런타임 타입을 가리키는 문자열을 반환하며, 타입스크립트 값과는 다르다.
    

- 타입스크립트의 타입 값의 종류는 무수히 많은 반면, 자바스크립트는 string, number, boolean, undefined, object, function 6가지의 런타임 타입만이 존재한다.

- this - 값으로 쓰이는 this는 자바스크립트의 this와 타입의 ‘다형성 this’라고 불리는 타입스크립트 타입으로 구분된다.

- &, | - 값 에서는 AND,OR 비트 연산자이며, 타입에서는 인터섹션과 유니온으로 구분된다.

- const - 값 에서는 새 변수를 선언하지만, as const는 리터럴 또는 리터럴 표현식의 투론된 타입을 바꾼다.

  

  

### Item 9. 타입 단언보다는 타입 선언 사용하기

- 타입 선언은 할당되는 값이 해당 인터페이스를 만족하는지 검사한다.

- 타입 단언(as ~)은 강제로 타입을 지정했으니 타입 체커에게 오류를 무시하라고 하는 것이다.

- 단언문은 꼭 필요한 경우가 아니라면, 권장되지는 않으며, 선언문을 사용하는 것이 좋다.

- const people**: Person[]** = [’a’, ‘b’, ‘c’].map(**(name): Person** ⇒ ({name}));
    
    - name의 타입은 Person[] 이 된다.
    
    - 위와 같이, 화살표 함수 내부에서 타입과 함께 변수를 선언하는 것이 좋다.
    
    - (name): Person은 name의 타입이 없고, 반환 타입이 Person이라고 명시한다.
    
    - const people: Person[] 에서는 함수의 반환 타입을 명시한다.
    

- 타입 단언이 필요한 경우도 있다.
    
    - 타입 체커보다 사용자가 더 정확한 타입 판단을 할 경우
        
        - 타입스크립트는 DOM에 접근할 수 없기 때문에 DOM 엘리먼트의 타입에 대해 정의할 때는 단언문을 사용하는 것이 더 좋을 수 있다.
        
    
    - null이 아님을 단언하는 경우, 변수 접미사 !를 사용해 null 이 아니라는 것을 단언할 수 있다.
    

  

  

  

### Item 10. 타입에 객체 래퍼 타입 피하기

- 자바스크립트 기본형 값들은 불변(immutable)이며 메서드를 가지지 않는 점에서 객체와 구분되지만, 기본형인 string에서 메서드를 가지고 있는것 처럼 보일 수 있다.
    
    - ‘primitive’.charAt(3) == “m”
    
    - 하지만 위의 경우, charAt은 string의 메서드가 아니다.
    
    - string 기본형에는 메서드가 없지만, 메서드를 가지는 String ‘객체’ 타입이 정의되어 있다.
    
    - 자바스크립트는 기본형과 객체형을 자유롭게 변환하며, 기본형에 메서드를 사용하고자 할 때, 기본형을 String과 같은 객체형으로 래핑(wrap)하고, 메서드를 호출한 후, 래핑한 객체를 버리는 과정을 거친다.
    

- 메서드 내의 this 객체는 기본형이 아닌 래퍼 객체를 가리킨다.

- 기본형과 래퍼 객체가 항상 동일하게 동작하는 것은 아니다.
    
    - String 객체는 오직 자기 자신하고만 동일하다.
    
    - 어떤 속성을 기본형에 할당하면 그 속성이 사라진다. 기본형을 객체 래퍼로 감싼 후 속성을 추가하고, 해당 속성이 추가된 객체는 버려지기 때문이다.
    

- number, boolean, symbol, bigint 에도 객체 래퍼 타입이 존재한다.

- 이 래퍼 타입 덕분에 기본형 값에 메서드를 사용할 수 있지만,

- 보통은 래퍼 객체를 직접 생성할 필요가 없다.
    
    - 기본형과 래퍼 객체 스펠링 오류 혹은 혼돈을 불러 일으킬 가능성이 있다.
    
    - string은 String에 할당할 수 있지만, String은 string에 할당할 수 없다.
    

  

  

  

### Item 11. 잉여 속성 체크의 한계 인지하기

- 타입이 명시된 변수에 객체 리터럴을 할당할 때, 타입스크립트는 해당 타입의 속성이 있는지, 그리고 ‘그 외의 속성은 없는지’ 확인한다.

- 잉여 속성 체크의 경우, 조건에 따라 동작하지 않는 경우가 있고,

- 통상적인 할당 가능 검사와 함께 쓰이면 구조적 타이핑이 무엇인지 혼란스러워질 가능성이 있다.

- 잉여 속성 체크는 할당 가능 검사와는 별도의 과정이라는 것을 알아야 타입 시스템에 대한 개념을 정확하게 잡을 수 있다.

- 타입스크립트는 단순히 런타임에 예외를 던지는 코드에 오류를 표시하는 것 뿐 아니라, 의도와 다르게 작성된 코드까지 찾으려고 한다.

- 잉여 속성 체크를 이용하면 기본적으로 타입 시스템의 구조적 본질을 해치지 않으면서도 객체 리터럴에 알 수 없는 속성을 허용하지 않음으로써 엄격한 리터럴 체크를 가능하게 한다.

- document나 HTMPAncorElement는 객체 리터럴이 아니기 때문에 잉여 속성 체크가 되지 않는다.

- {a:’aa’, b:’bb’} 와 같은 객체 리터럴에서는 잉여 속성체크르 사용할 수 있지만, 일반 문자열과 같은 객체 리터럴이 아닌 경우에는 잉여 속성 체크가 사용되지 않고 잉여 속성 체크로 인한 오류는 사라지게 된다.

- 잉여 속성체크는 타입 단언문을 사용할 때에도 적용되지 않는다. (선언문 권장되는 이유)

- 잉여 속성 체크를 원하지 않는다면, interface 내부 [otherOpts: string]: unknown; 과 같은 인덱스 시그니처를 사용해 타입스크립트가 추가적인 속성을 예상하도록 할 수 있다.

- 모든 속성이 선택적 옵셔널 값인 ‘약한 타입’에 대해서 타입스크립트는 값 타입과 선언 타입에 공통된 속성이 있는지 확인하는 별도의 체크를 수행한다. 그러나 잉여 속성 체크와는 다르게 약한 타입과 관련된 할당문마다 실행된다. 임시 변수를 제거하더라도 공통 속성 체크는 여전히 동작한다.

- 잉여 속성 체크는 구조적 타이핑 시스템에서 허용되는 속성 이름의 오타 같은 실수를 잡는데 효과적인 방법이다
    
    - 선택적 필드를 포함하는 타입에 특히 유용한 반면, 적용 범위도 매우 제한적이고, 오직 객체 리터럴에만 적용된다.
    

  

  

  

### Item 12. 함수 표현식에 타입 적용하기

- 자바/타입 스크립트에서는 함수 표현식과 선언식을 다르게 인식한다.

- 타입 스크립트에서는 함수 표현식을 사용하는 것이 좋다. 함수 매개변수부터 반환 값까지 전체를 함수 타입으로 선언해 함수 표현식에 재 사용할 수 있기 때문이다.

- 시그니처가 일치하는 다른 함수가 있을 때, 함수 표현식에 타입을 적용해 볼 만 하다.

- 함수의 매개변수에 타입 선언을 하는 것보단 함수 표현식 전체 타입을 정의하는 것이 코드도 간결하고 안전하다.

- 다른 함수의 시그니처와 동일한 타입을 가지는 새 함수를 작성하거나 동일한 타입 시그니처를 가지는 여러개의 함수를 작성할 때는 매개변수의 타입과 반환 타입을 반복해서 작성하지 말고 함수 전체의 타입 선언을 적용해야 한다.

- 다른 함수의 시그니처를 참조하려면 typeof fn을 사용하면 된다.

  

  

  

### Item 13. 타입과 인터페이스 차이점 알기

- 대부분의 경우, 타입과 인터페이스 어떤 것을 사용해도 된다.

- 명명된 타입은 인터페이스로 정의하든, 타입으로 정의하든 상태에는 차이가 없다.

- 유사점
    
    - 인덱스 시그니처는 인터페이스와 타입 모두에서 사용할 수 있다.
    
    - 함수 타입 또한 인터페이스와 타입 모두에서 지정할 수 있다.
    
    - 타입 별칭과 인터페이스 모두 제너릭이 가능하다.
    
    - 인터페이스는 타입을 확장할 수 있으며, 타입은 인터페이스를 확장할 수 있다.
        
        - 인터페이스는 유니온과 같은 복잡한 타입을 확장하지는 못한다. 복잡한 타입을 확장하고 싶다면, 타입과 &를 사용해야 한다.
        
    
    - 클래스 구현시(implements)에도 타입과 인터페이스 모두 사용할 수 있다.
    

- 차이
    
    - 유니온 타입은 있지만 유니온 인터페이스 개념은 없다.
        
        - 인터페이스는 타입을 확장할 수 있지만, 유니온은 할 수 없다.
        
        - 유니온 타입을 확장하는 일이 필요할 수 있다.
        
    
    - 타입 키워드는 일반적으로 인터페이스보다 쓰임새가 많다. 타입 키워드는 유니온이 될 수도 있고, 매핑된 타입 또는 조건부 타입 같은 고급 기능에 활용되기도 한다.
    
    - 튜플과 배열 타입도 타입 키워드를 이용해 더 간단하게 표현 가능
        
        - type StringList = string[];
        
        - type NamedNums = [string, …number[]];
        
        - 인터페이스도 튜플과 비슷하게 구현하면 사용은 할 수 있지만, 튜플에서 사용할 수 있는 concat 같은 메서드들을 사용할 수 없게 된다.
        
        - 그러므로 튜플은 타입 키워드로 구현하는 것이 더 낫다
        
    
    - 인터페이스는 타입 키워드에는 없는 ‘보강’이 가능하다
        
        - 동일한 인터페이스를 한번 더 생성해 두 가지의 인터페이스가 보강되어 각각 인터페이스 속성들을 하나로 통합하는 기능
        
        - 이를 ‘선언 병합’이라고 하며, 타입 선언 파일을 작성할 때는 선언 병합을 지원하기 위해 반드시 인터페이스를 사용해야 하며, 표준을 따라야 한다.
        
        - 타입스크립트는 여러 버전의 자바스크립트 표준 라이브러리에서 여러 타입을 모아 병합한다. 여러 버전의 타입 선언들이 병합되어 전체 메서드를 가지는 하나의 Array 타입을 얻게 된다.
        
        - 병합은 선언처럼 일반적인 코드이기 때문에 언제든 가능하며, 프로퍼티가 더 이상 추가되는 것을 원하지 않는다면 인터페이스 보다는 타입을 사용해야 한다.
        
    
    - 복잡한 타입이라면 타입 키워드를 사용하는 것이 좋고,
    
    - 타입과 인터페이스 두 가지 모두로 표현 가능한 간단한 객체 타입이라면 일관성과 보강의 관점에서 고려해야 한다.
    
    - 기존 코드 베이스가 무엇을 사용했는지에 따라서도 나뉠 수 있다.
    

  

  

  

### Item 14. 타입 연산과 제너릭 사용으로 반복 줄이기

- 반복을 줄이는 가장 간단한 방법은 타입에 이름을 붙이는 것이다.

- 인터페이스 확장을 통한 반복 제거 가능

- 부분집합으로 타입 정의
    
    ```jsx
    interface State {
    	userId: string;
    	pageTitle: string;
    	recentFiles: string[];
    	pageContents: string;
    }
    
    interface TopNavState {
    	userId: string;
    	pageTitle: string;
    	recentFiles: string[];
    }
    
    // State의 부분집합으로 TopNavState 정의
    type TopNavState = {
    	userId: State['userId'];
    	pageTitle: State['pageTitle'];
    	recentFiles: State['recentFiles'];
    }
    
    // 위 방식을 매핑된 타입을 사용해 변경
    type TopNavStatus = {
    	[k in 'userId' | 'pageTitle' | 'recnetFiles']: State[k];
    }
    
    // Pick
    type Pick<T, K> = { [k in K]: T[k] };
    ```
    
    - 매핑된 타입은 배열의 필드를 루프 도는 것과 같은 방식 - Pick(표준 라이브러리) 이다.
    
    - 위에서 Pick은 제너릭 타입이다. 중복된 코드를 없앤다는 관점으로, Pick을 사용하는 것은 함수를 호출하는 것에 비유할 수 있다. 마치 함수에서 두 개의 매개변수 값을 받아서 결과값을 반환하는 것처럼, Pick은 T와 K 두 가지 타입을 받아서 결과 타입을 반환한다.
    

  

  

  

### Item 15. 동적 데이터에 인덱스 시그니처 사용하기

```tsx
interface Row { a: number; b: number; c?:number };
```

- 너무 광범위한 string, number 타입을 대체할 만한 수단으로
    
    ```tsx
    // Record : 키 타입에 유연성을 제공하는 제너릭, string의 부분 집합을 사용할 수 있다.
    type Vec3D = Recore<'x' | 'y' | 'z', number>;
    // 매핑된 타입: 키마다 별도의 타입을 사용하게 해준다
    type Vec3D = { [k in 'x' | 'y' | 'z']: number}
    // Type Vec3D = {
    //    x: number;
    //    y: number;
    //    x: number;
    // }
    // 위 두 방법의 결과는 동일하다.
    
    // 매핑된 타입 추가 예제
    type ABC = {[ k in 'a' | 'b' | 'c']: k extends 'b'? string : number};
    // Type ABC = {
    //    a: number;
    //    b: string;
    //    c: number;
    // }
    ```
    

- 런타임 때 까지 객체의 속성을 알 수 없는 경우(ex: CSV 파일에서 로드하는 경우 등) 에만 인덱스 시그니처를 사용하는 것이 좋다

- 가능하다면, 인터페이스, Record, 매핑된 타입과 같은 정확한 타입을 사용하는 것이 좋다.

  

  

  

### Item 16. number 인덱스 시그니처보다는 Array, 튜플, Arraylike 사용하기

- 자바스크립트에서 객체란, 키와 값 쌍을 의미한다. 키는 보통 문자열이며, 값은 어떤 것이든 올 수 있다.

- 복잡한 값, 숫자 등을 키로 사용하려 하면, 자바스크립트는 키 값을 문자열로 변환한다.
    
    - x = [1, 2, 3]; ⇒ x[’1’] 로도 접근이 가능하다.
    

- 타입스크립트는 이러한 동작을 바로잡기 위해 숫자 키를 허용하고 문자열 키와 다른 것으로 인식한다. ( lib.es5.d.ts 에서 확인 )

- 배열의 순회에서, index를 신경쓰지 않는다면, for-of 순회를 사용하는 것이 좋다.

- 인덱스의 타입이 중요하다면, forEach를 사용하는 것이 좋다.

- 루프 중간에 멈춰야 한다면, C스타일의 for(;;) 순회를 사용하는 것이 좋다.

- 타입이 불확실하다면, 대부분의 브라우저, 자바스크립트 엔진에서는 for-in 순회가 for-of 혹은 for(;;) 순회에 비해 몇 배나 느리다.

- 인덱스 시그니처가 number로 표현되어 있다면, 입력한 값이 number이어야 한다는 것을 의미하지만, 실제 런타임에 사용되는 키는 string이다.
    
    - 일반적으로 string 보다 number를 타입의 인덱스 시그니처로 사용할 이유는 많지 않다.
    
    - 만약 숫자를 사용해 인덱스 할 항목을 지정한다면, Array 또는 튜플 타입을 대신 사용하게 될 것이다.
    
    - number 타입을 인덱스 타입으로 사용하게 된다면, 숫자 속성이 어떤 특별한 의미를 지닌다는 오해를 불러 일으킬 수 있다.
    

- 어떤 길이를 가지는 배열과 비슷한 형태의 튜플을 사용하고 싶다면, 타입스크립트에 있는 ArrayLike 타입을 사용한다.
    
    ```tsx
    funtion a<T>(xs: ArrayLike<T>, i: number): T {
    	if(i < xs.length){
    		return xs[i];
    	{
    }
    ```
    
    - ArrayLike를 사용해도 키 값은 항상 문자열이다.
    

  

  

### Item 17. 변경 관련된 오류 방지를 위해 readonly 사용하기

- readonly number[]는 타입이지만, number[]와 구분되는 몇 가지 특징이 있다.
    
    - 배열의 요소를 읽을 순 있지만, 쓸 수는 없다.
    
    - length를 읽을 수 있지만, 바꿀 순 없다.
    
    - 배열을 변경하는 pop을 비롯한 다른 메서드를 호출할 수 없다.
    

- number[]는 readonly number[]보다 기능이 많기 때문에 readonly number[]의 서브 타입이 된다.
    
    - 따라서, 변경 가능한 배열을 readonly배열에 할당할 수 있지만, 그 반대는 불가능하다.
    

- 자바스크립트, 타입스크립트에서는 명시적으로 언급하지 않는 한, 함수가 매개변수를 변경하지 않는다고 가정한다.
    
    - 하지만 이러한 암묵적 방법은 타입 체크에 문제를 일으킬 수 있다.
    
    - 명시적인 방법을 사용하는 것이 사람과 컴파일러 모두에게 좋다.
    

- 만약 함수가 매개변수를 변경하지 않는다면, readonly로 선언해야 한다. 더 넓은 타입으로 호출할 수 있고, 의도치 않은 변경은 방지할 수 있기 때문이다.

- 이로 인한 단점으로는,
    
    - 매개변수가 readonly로 선언되지 않은 함수를 호출해야 하는 경우도 있다는 것이다.
    
    - 만약 라이브러리에 있는 readonly 함수를 호출하는 경우라면, 타입 단언문(as ~)을 사용해야 한다.
    

- readonly를 사용하면 지역 변수와 관련된 모든 종류의 변경 오류를 방지할 수 있다.

- readonly는 얕게(shallow) 동작하다는 것에 유의하며 사용해야 한다. 만약 깊은 readonly를 사용하고자 한다면, 라이브러리를 사용하는 것이 좋다.

  

  

  

### Item 18. 매핑된 타입을 사용해 값을 동기화하기

- 매핑된 타입은 한 객체가 또 다른 객체와 정확히 같은 속성을 가지게 할 때 이상적이다.

```tsx
interface types {
	a: number[];
	b: number[];
	...
}

const A: (keyof types)[] = [
	'a',
	'b'
	...
]
```

- 매핑된 타입을 사용해 타입스크립트가 코드에 제약을 강제하도록 할 수 있다.

- 매핑된 타입을 사용해 관련된 값과 타입을 동기화한다.

- 인터페이스에 새로운 속성을 추가할 때, 선택을 강제하도록 매핑된 타입을 고려해야 한다.
