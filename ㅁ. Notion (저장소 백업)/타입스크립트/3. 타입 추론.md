### Item 19. 추론 가능한 타입을 사용해 장황한 코드 방지하기

- 사실상, 타입스크립트의 많은 타입 구문은 불필요하다.

- 타입 추론이 된다면 명시적 타입 구문은 필요하지 않다.

- 타입스크립트는 단순 원시 변수에 대한 타입 추론 뿐만 아니라, 복잡한 객체도 추론할 수 있다.

- 입력을 받아 연산을 하는 함수가 반환하는 타입 또한 추론이 가능하다.

- const { id, name, price } = product; 와 같은 비구조화 할당문은 모든 지역 변수의 타입이 추론되도록 한다. 그렇기 때문에 추가로 명시적 타입 구문을 추가하게 되면, 코드가 번잡해진다.

- 정보가 부족해서 타입스크립트 스스로 타입을 판단하기 어려운 상황일 때, 명시적 타입 구문이 필요하다.
    
    - 예를 들어, 함수의 매겨변수로 들어오는 값의 타입을 지정할 때가 있다.
    

- 타입스크립트에서 변수의 타입은 일반적으로 처음 등장할 때 정해진다.

- 이상적인 타입스크립트 코드는 함수/메서드 시그니처에 타입 구문을 포함하지만, 함수 내에서 생성된 지역 변수에는 타입 구문을 넣지 않는다.

- 매개 변수의 기본 값을 설정해 놓은 경우에도 타입 추론이 되기 때문에 따로 명시적으로 타입 구문을 추가하지 않는다.

- 보통, 타입 정보가 있는 라이브러리에서 콜백함수의 매개변수 타입은 자동으로 추론된다. 예를 들면, express HTTp 서버 라이브러리를 사용하는 request와 response의 타입 선언은 필요하지 않다.

- 타입 추론이 가능함에도 불구하고, 명시적인 타입 구문을 추가하는 상황은 몇 가지가 있는데,
    
    - 객체 리터럴을 정의할 때 타입을 명시하면 ‘잉여 속성 체크’가 동작한다. ‘잉여 속성 체크’는 특히 선택적 속성이 있는 타입의 오타 같은 오류를 잡는데 효과적이다. 또한, 변수가 사용되는 시점이 아닌, 할당하는 시점에 오류를 표시 되도록 해준다.
    
    - 만약, 타입 구문을 제거한다면 잉여 속성 체크가 동작하지 않고, 객체를 선언한 곳이 아닌 객체가 사용되는 곳에서 타입 오류가 발생한다.
    
    - 또한, 함수의 반환에도 타입을 명시해 오류를 방지할 수 있다. 반환 타입을 명시하면 구현상의 오류가 사용자 코드의 오류로 표시되지 않는다. ( Promise와 관련된 오류를 피하는 데는 async 함수가 효과적 )
    
    - 반환 타입을 명시해야 하는 추가적인 이유로,
        
        1. 함수에 대해 명확하게 알 수 있으며,
        
        1. 명명된 타입을 사용하기 위함이다.
        
        반환 타입을 명시하면 더욱 직관적인 표현이 된다. 또한, 반환 값을 별도의 타입으로 정의하면 타입에 대한 주석을 작성할 수 있어서 더욱 자세한 설명이 가능하다.
        
    

  

  

  

### Item 20. 다른 타입에는 다른 변수 사용하기

- 변수의 값은 바뀔 수 있지만, 그 타입은 보통 바뀌지 않는다.

- 타입을 바꿀 수 있는 한 가지 방법은 범위를 좁히는 것인데, 새로운 변수값을 포함하도록 확장하는 것이 아니라 타입을 더 작게 제한하는 것이다.

- 변수를 무분별하게 재사용하는 것은 타입 체커와 사람 모두에게 혼란을 줄 수 있다. 다른 타입에는 다른 변수를 사용하는 방법에서의 이점은
    
    - 서로 관련이 없는 두 값 분리
    
    - 변수명 구체화
    
    - 타입 추론 향상, 타입 구문 불필요
    
    - 타입 간결화
    
    - const로 변수 선언, const로 변수를 선언하면 코드가 간결해지고, 타입 추론도 더 정확해진다.
    

  

  

  

### Item 21. 타입 넓히기

- 런타임에 모든 변수는 유일한 값을 가진다.

- 하지만, 타입스크립트가 작성된 코드를 체크하는 정적 분석 시점에 변수는 ‘가능한’ 값들의 집합인 타입을 가진다.

- 상수를 사용해 변수를 초기화할 때, 타입을 명시하지 않으면 타입 체커는 타입을 결정해야 한다. 이는 지정된 단일 값을 가지고 할당 가능한 값들의 집합을 유추해야 한다는 뜻이다.

- 위 과정을 타입스크립트에서는 ‘타입 넓히기’ 라고 칭한다.

- 타입 넓히기가 진행될 때, 주어진 값으로 추론 가능한 타입이 여러 개이기 때문에 과정이 상당히 모호해진다. 정보가 충분하지 않다면 해당 값이 어떤 타입으로 추론되어야 하는지 알 수 없다.
    
    - 그러므로 타입스크립트는 작성자의 의도를 추측한다.
    

- 타입스크립트는 넓히기 과정을 제어할 수 있도록 몇 가지 방법을 제공한다.
    
    - 넓히기 과정을 제어하는 첫 번째는 const이다. let 대신 const를 사용하면 더 좁은 타입이 된다.
    
    - 하지만 const도 객체와 배열의 경우에 만능은 아니다. 객체의 경우, 타입스크립트 넓히기 알고리즘은 각 요소를 let으로 할당된 것 처럼 다룬다.
    
    - 타입 추론의 강도를 직접 제어하려면 타입스크립트의 기본 동작을 재정의해야 한다.
        
        1. 명시적 타입 구문을 제공
        
        1. 타입 체커에 추가적인 문맥을 제공 ( 함수의 매개변수로 값을 전달 등의 방법을 통해 )
        
        1. const 단언문을 사용한다. 변수 선언에 사용되는 const와는 다르며, const 단언문은 완전히 타입 공간의 기법이다. 값 뒤에 as const를 사용하면 타입스크립트는 최대한 좁은 타입으로 추론한다. 배열을 튜플 타입으로 추론할 때에도 as const를 사용할 수 있다.
        
    

  

  

  

### Item 22. 타입 좁히기

- 타입스크립트가 넓은 타입에서 좁은 타입으로 진행하는 과정을 의미

- 가장 일반적 예시는 null 체크, 분기문을 사용해 null 타입 가능성을 제거하는 방식으로, 이러한 방식은 타입 별칭이 존재하면 사용하지 못할 수 있다.

- 분기문에서 예외를 던지거나 함수를 반환해 블록의 나머지 부분에서 변수의 타입을 좁힐 수도 있다.

- instanceof 를 사용해 타입을 좁히는 방법도 있다.

- 속성 체크를 사용하는 방법
    
    ```tsx
    interface A { a: number };
    interface B { b: number };
    
    pickAb = (ab: A | B)=>{
    	if('a' in ab){
    		ab    // 타입이 A
    	}else if('b' in ab){
    		ab    // 타입이 B
    	}
    	ab    // 타입이 A | B
    }
    ```
    

- Array.isArray와 같은 내장함수를 사용해서도 타입을 좁힐 수 있다.

- 타입스크립트는 일반적으로 조건문에서 타입을 좁히는데 매우 능숙하다.

- 그러나 타입을 섣불리 판단하는 실수를 할 수 있다.
    
    - 자바스크립트에서 typeof null이 ‘object’ 이기 때문에, if 구문애서 null이 제외되지 않는 경우가 발생할 수 있다.
    
    - 빈 문자열 ‘’ 혹은 0 이 false이기 때문에 타입이 좁혀지지 않는 경우가 발생할 수 있다.
    

- ‘태그’를 사용해 타입을 좁힐 수도 있다. ( 태그된 유니온 or 구별된 유니온 )
    
    ```tsx
    interface UploadEvent { type: 'upload'; ~~ };
    interface DownloadEvent { type: 'download'; ~~ };
    
    function (e)~~
    switch ( e.type ):
      case: 'download':
      	e    // DownloadEvent 타입
      	break;
      case 'upload':
    	  e    // UploadEvent 타입
    	  break;
    ~~
    ```
    

- 사용자 정의 타입 가드 사용 ( 타입스크립트가 타입을 식별하지 못할 경우, 식별을 돕기 위해 커스텀 함수 도입 )

  

  

  

### Item 23. 한꺼번에 객체 생성하기

- 변수의 값을 변경될 수 있지만, 타입스크립트의 값은 일반적으로 변경되지 않는다.

- 객체를 생성할 때는 속성을 하나씩 추가하기보다는 여러 속성을 포함해 한꺼번에 생성해야 타입 추론에 유리하다. ( 객체 속성 한번에 미리 정의해놓으면 타입 선언 필요 X, 추론 가능 )

- 반드시 객체를 제각각 나눠서 만들어야 한다면, 타입 단언문(as)을 사용해 타입 체커를 통과하게 할 수 있다.
    
    ```tsx
    cosnt pt = {} as Point;
    pt.x = 3;
    pt.y = 4;
    ```
    
    - 물론, 위의 경우에도 가능하면 객체를 한번에 선언하는 방식이 더 좋다.
    

- 작은 객체들을 조합해서 큰 객체를 만들어야 하는 경우에도 여러 단계를 거치는 것은 좋은 방법이 아니다.
    
    - 위의 경우, const aa = {…pt, …id} 와 같이 객체 전개 연산자를 사용하면 큰 객체를 한번에 만들어낼 수 있다.
    
    - 객체 전개 연산자를 사용하면 타입 걱정 없이 필드 단위로 객체를 생성할 수 있다.
    
    - 타입에 안전한 방식으로 조건부 속성을 추가하려면, 속성을 추가하지 않는 null 또는 {} 으로 객체 전개 연산자를 사용하면 된다.
    
    ```tsx
    const aaa = { ...first, ...(second ? {third: 'S'} : {} ) };
    ```
    

- 유니온을 사용하는게 가능한 값의 집합을 더 정확히 표현항 수 있지만, 유니온보다는 선택적 필드가 다루기는 더 쉬울 수 있다. 선택적 필드 방식으로 표현하려면 아래처럼 헬퍼 함수를 사용하면 된다
    
    ```tsx
    function addOptional<T extends object, U extends object>(
    a: T, b: U | null
    ) : T & Partial<U> {
    	return { ...a, ...b };
    }
    
    const pharaoh = addOptional(
    	nameTitle,
    	hasDates ? { start: -2598, end: -2566 } : null
    );
    
    pharaoh.start    // 정상, 타입이 number | undefined
    ```
    

  

  

  

### Item 24. 일관성 있는 별칭 사용하기

- 별칭은 타입스크립트가 타입을 좁히는 것을 방해한다. 따라서 변수에 별칭을 사용할 때는 일관되게 사용해야 한다.

- 비구조화 문법 const { x, y } = obj; 을 사용해 일관된 이름을 사용하는 것이 좋다.

- 함수 호출이 객체 속성의 타입 정제를 무효화할 수 있다는 점을 주의해야 한다. 속성보다 지역 변수를 사용하면 타입 정제를 믿을 수 있다.

  

  

  

### Item 25. 비동기 코드에는 콜백 대신 async 함수 사용하기

- 콜백 함수를 연속으로 사용하는 경우, 실행의 순서는 코드의 순서와 반대가 된다.

- 이러한 콜백 중첩 코드는 직관적으로 이해하기 어렵다.

- 요청들을 병렬로 실행하거나 오류 상황을 빠져나오고 싶다면 더욱 혼란스러워진다.

- 이러한 문제점을 극복하기 위해 Promise 개념을 도입

- 더욱 간단하게 처리할 수 있도록 async / await 기법을 도입
    
    - await 키워드는 각각의 프로미스가 처리(resolve)될 때 까지 fetchPages 함수의 실행을 멈춘다.
    
    - async 함수 내에서 await 중인 프로미스가 거절(reject) 되면 예외를 던진다.
    
    - 이를 통해 일반적은 try/catch 구문을 사용할 수 있게 된다.
    

- 타입스크립트는 런타임에 관계 없이 async/await 를 사용할 수 있다.
    
    - 예를 들어, 병렬로 페이지를 로드하고 싶은 경우, Promise.all을 사용해 프로미스를 조합할 수 있다. 이 경우, await와 구조 분해 할당이 잘 맞는다.
    
    ```tsx
    async function fetchPages(){
    	const [ response1, response2, response3 ] = await Promise.all([
    		fetch(url1), fetch(url2), fetch(url3),
    	]);
    }
    ```
    
    - 타입스크립트는 세 가지 response 변수 각각의 타입을 Response로 추론한다.
    
      
    
    - 입력된 프로미스들 중 첫 번째가 처리될 때 완료되는 Promise.race도 타입 추론과 잘 맞는다
        
        - Promise.race를 사용해 프로미스에 타임아웃을 추가하는 방법은 흔하게 사용되는 패턴이다.
        
    
    ```tsx
    function timeout( miles: number ): Promise<never> {
    	return new Promise(( resolve, reject ){
    		setTimeout(() => reject('timeout'), miles);
    	});
    };
    
    async function fetchWithTimeout (url: string, ms: number){
    	return Promise.race([fetch(url), timeout(ms)]);
    }
    ```
    
    - 타입 구문 없이도 fetchWithtimeout의 반환 타입은 Promise<Response>로 추론되는데, 이는 Promise.race의 반환 타입은 입력 타입들의 유니온이기 때문에 위의 경우, Promise<Response | never>가 된다. 그러나 never(공집합)와 유니온은 아무런 효과가 없으므로 결과가 Promise<Response>로 간단해지기 때문이다.
    

- 프로미스를 사용할 경우, 타입스크립트의 모든 타입 추론이 제대로 동작한다.

- 가끔 프로미스를 직접 생성해야 하는 경우가 있는데, setTimeout과 같은 콜백 API를 래핑하는 경우 같은 때가 아닌, 선택지가 있는 경우라면 일반적으로는 Promise를 직접 생성하기 보다는 async/await를 사용해야 한다.
    
    - 이유는, 간격하고, 직관적이며, async 함수는 항상 프로미스를 반환하도록 강제되기 때문이다.
    

- 함수는 항상 동기 또는 비동기로 실행되어야 하며, 프로미스는 함수를 항상 비동기 함수로 통일하도록 강제하는데 도움이 된다.
    
    - 프로미스를 사용하는 경우, 실수로 반쪽짜리 동기 코드를 사용할 수 도 있지만, async 를 사용하는 경우 비동기가 강제되기 때문에 항상 비동기 코드를 작성하게 된다.
    

- async 함수에서 프로미스를 반환하면 또 다른 프로미스로 래핑되지 않는다. 반환 타입은 Promise<Promise<T>>가 아닌 Promise<T> 가 된다.

  

  

  

### Item 26. 타입 추론에 문맥이 어떻게 사용되는지 이해하기

- 타입스크립트가 타입을 추론할 때, 단순히 값 뿐만 아니라 문맥까지 고려한다.

- 문자열 리터럴 타입의 유니온을 받는 함수에 유니온 값에 해당하는 문자열을 변수에 할당해 변수를 매개변수로 지정하면 유니온에 해당하는 값이더라도 타입 에러가 출력된다.
    
    - 타입스크립트는 할당 시점에 타입을 추론한다. ( 타입스크립트는 일반적으로 값이 처음 등장할 때 타입을 결정한다. )
    
    - 위의 경우, 유니온 리터럴 값을 변수에 할당하는 시점에 타입 추론이 되기 때문에 유니온 리터럴 값이 아닌, 단순 string 값으로 추론이 되게 된다.
    
    - 위 문제를 해결하기 위해서는 두 방법이 있다.
        
        - 타입 선언에서 가능한 타입 값을 제한하는 방법 ( 타입 지정 )
        
        - 해당 변수를 const(상수)로 선언해 해당 변수는 변경될 수 없는 값이라는 것을 타입 체커에게 알려주어 타입 체커는 string 보다는 더 정확한 타입인 리터럴 유니온 타입으로 추론할 수 있게 된다.
        
    
    - 문맥으로부터 값을 분리했을 경우의 문제
        
        - 튜플 사용시
            
            - 타입 선언 제공
            
            - 상수 문맥 제공(as const) - const 는 단지 값이 가리키는 참조가 변하지 않는 얕은 상수인 반면, as const 는 그 값이 내부까지 상수라는 사실을 타입스크립트에게 알려준다. 하지만, 이 방법은 너무 과하게 정확하다. 또한, 타입 정의에 실수가 있다면, 오류는 타입 정의가 아니라 호출되는 곳에서 발생한다. 이는 여러 겹 중첩된 객체에서 어류가 발생한다면 근본적인 원인 파악을 어렵게 한다.
            
            - readonly 구문 추가
            
        
        - 콜백 사용시
            
            - 콜백을 다른 함수로 전달할 때, 타입스크립트는 콜백의 매개변수 타입을 추론하기 위해 문맥을 사용한다.
            
            - 콜백을 상수로 뽑아내면 문맥이 소실되고 noInplicitAny 오류가 발생한다.
            
            - 이 경우는 매개변수에 타입 구문을 추가해 해결할 수 있다.
            
            - 가능할 경우, 전체 함수 표현식에 타입 선언을 적용한다.
            
        
    

  

  

  

### Item 27. 함수형 기법과 라이브러리로 타입 흐름 유지하기

- 라이브러리들의 일부 기능(map, flatMap, filter 등)은 타입스크립트와 조합해 사용하면 더욱 좋다.

- 타입 정보가 유지되면서 타입 흐름이 계속 전달되도록 하기 때문이다.

- 반면 루프를 직접 구현하면 타입 체크에 대한 관리도 직접 해야 한다.

- 같은 코드를 타입스크립트로 작성한다면 서드파티 라이브러리를 사용하는것이 무조건 유리하다. 타입 정보를 참고하며 작업할 수 있기 때문에 서드파티 라이브러리 기반으로 바꾸는데 시간이 훨씬 단축된다.

- 루프를 사용해 단순(flat) 목록을 만들려면 배열에 concat 을 사용해야 한다. 이 경우, 코드는 동작되지만, 타입 체크는 되지 않는다.
    
    - 이 오류를 고치려면, 타입 구문을 추가해야 한다.
    
    - 그러나 Array.prototype.flat() 을 사용하면 더욱 간단하게 해결할 수 있다.
    
    - flat 메서드는 다차원 배열을 평탄화(flatten) 해준다.
    
    - 타입 시그니처는 T[][] ⇒ T[] 같은 형태이며, 이 버전이 가장 간결하고 타입 구문도 필요 없다.
    

- 로대시를 사용하면 체인 개념을 사용하는데, 이는 일련의 순서로 연산을 작성할 수 있도록 해준다.
    
    - 만약 체인을 사용하지 않는다면, _.c(_.b(_a(v))) 와 같이 뒤에서부터 연산이 된다.
    
    - 체인을 사용하면 _(v).a().b().c().value() 로 연산자의 등장 순서와 실행 순서가 동일해진다.
    
    - _(v)는 값을 래핑하고, value()는 값을 언래핑한다.
    
    - _(v)는 값을 래핑하고, value()는 값을 언래핑한다.
    

- 타입스크립트는 함수 호출 시 잘 전달된 매개변수 값을 건드리지 않고 매번 새로운 값을 반환함으로서 새로운 타입으로 안전하게 반환할 수 있다.
    
    - 타입스크립트의 많은 부분이 자바스크립트 라이브러리의 동작을 정확히 모델링하기 위해 개발되었기 때문에 라이브러리를 사용할 때 타입 정보가 잘 유지되는 점을 이용해야 한다.
