## 7.1 여러 컨테이너 관리의 개요

### 웹 3계층 시스템 아키텍처

> [!important]
> 
> **프론트 서버**
> 
> 브라우저(클라이언트) ==<——HTTP요청/응답——>== 웹 프론트 서버(=웹 서버)
> 
> _==사용자 인터페이스(UI)를 제공하고, 사용자의 입력을 처리하여 애플리케이션 서버로 전달==_
> 
> - 예시: 미들웨어로 구축, Nginx, IIS, Apache 등

> [!important]
> 
> **애플리케이션 서버**
> 
> 프론트 서버 ==<—— 비즈니스 로직 수행 ——>== 애플리케이션 서버
> 
> _==실제 비즈니스 로직을 처리하는 실행 환경. 클라이언트 요청을 처리하고, 데이터베이스와 상호작용하며, 비즈니스 로직을 수행==_
> 
> - 예시: FastAPI로 구현된 REST API 서버

> [!important]
> 
> **데이터베이스 서버**
> 
> 애플리케이션 서버 ==<——데이터 요청/응답——>== 데이터베이스 서버
> 
> _==데이터를 저장, 관리, 조회, 삽입, 수정, 삭제 등 수행==_
> 
> - 예시: MySQL, PostgreSQL, Oracle Database, Redis, MongoDB

  

- **3계층 아키텍처의 장점**
    
    - ✅ 유지보수 용이: 각 계층이 독립적이라 수정해도 다른 계층에 미치는 영향이 적음.
    
    - ✅ 확장성 (Scalability): 각 계층을 개별적으로 확장 가능 (예: DB 성능이 부족하면 DB만 확장).
    
    - ✅ 보안 강화: 클라이언트가 직접 데이터베이스에 접근하지 않고, API를 통해 접근하도록 제한 가능.
    
    - ✅ 재사용성 (Reusability): 같은 비즈니스 로직을 다양한 클라이언트(웹, 모바일)에서 재사용 가능.

docker-compose 정의 파일 확인

- compose 정의 파일 설명
    
    ```bash
    version: '3.3'  # Docker Compose 파일의 버전 지정
    
    services:  # 실행할 서비스 목록 정의
    
      # WebServer config
      webserver:  # 웹 서버 서비스 정의
        build: .  # 현재 디렉토리에서 Dockerfile을 찾아 이미지를 빌드
        ports:
         - "80:80"  # 호스트의 80번 포트를 컨테이너의 80번 포트에 매핑
        depends_on:
         - redis  # redis 서비스가 먼저 실행된 후 웹 서버가 실행됨
    
      # Redis config
      redis:  # Redis 서비스 정의
        image: redis:4.0  # Redis 4.0 버전의 공식 Docker 이미지를 사용하여 실행
    ```
    
    - `"80:80"`에서 **왼쪽 80**이 호스트(외부에서 접근하는 PC)의 포트/ **오른쪽 80**이 **컨테이너 내부의 포트**
        
        - `http://localhost:80` (호스트에서 접근)
        
        - 컨테이너 내부에서는 `http://0.0.0.0:80` 로 서비스 실행됨

`docker-compose.yml` : Docker Compose의 구성 관리 파일(YAML 형식)

- 여러 컨테이너의 설정 내용을 모아 하나의 파일에 기술

- 관리하고 싶은 컨테이너의 서비스, 네트워크, 볼륨을 정의

- `#` 으로 주석 표현

  

> [!important]
> 
> **YAML**
> 
> ---
> 
> - 구조화된 데이터를 표현하기 위한 데이터 포맷

> - 들여 쓰기(반각 스페이스)로 데이터 계층 구조 표현(Python과 유사)

> - 데이터의 맨 앞에 `-` 를 붙이면 배열 표현 가능 (각 항목 표현, 기호 뒤 스페이스 필수)
> 
>   
> 
> * 반각 스페이스: 일반적인 스페이스 공백을 의미

## **🌊 이미지 지정(image)**

- Docker 컨테이너의 바탕이 되는 베이스 이미지를 지정
    
    - 베이스 이미지가 로컬 환경에 있으면 그걸 사용하고, 없으면 Docker Hub로부터 자동 다운로드
    

- 이미지의 이름 또는 이미지 ID 중 하나를 지정

- 제품 환경에서 사용 시 태그를 지정해 버전을 고정하는 것을 권장.

```yaml
services:
webserver:
	image: asashiho/dockersample:1.0
```

## **🌊 이미지 빌드(build)**

- Dockerfile을 사용하여 이미지를 빌드할 때 `build` 속성 지정
    
    - `build`: Dockerfile이 위치한 디렉토리 경로 지정
    

  

1️⃣ 현재 디렉토리의 Dockerfile을 빌드 (`build: .`)

```yaml
services:
  webserver:
    build: .
```

`docker-compose up --build` 명령으로 컨테이너 실행

```docker
docker-compose up --build
```

`docker-compose.yml` 내에서 지정한 Dockerfile이 자동으로 빌드되어 컨테이너가 생성됨

  

2️⃣ 특정 Docker파일 지정

- `dockerfile` 지정 필요

- `context` 로 dockerfilie의 경로 지정

```yaml
services:
  webserver:
    build:
      context: /data  # Dockerfile이 있는 경로
      dockerfile: Dockerfile-alternate  # 사용할 Dockerfile의 이름
```

  

3️⃣ 빌드 시 인수(`args`) 지정

- 이미지 빌드 시 인수 지정 가능

- bool 연산자의 경우, 따옴표로 감싸야 함

- 해당 변수의 값은 Docker Compose를 실행하는 머신 위에서만 유효함

```yaml
services:
  webserver:
    build:
      args:
        projectno: 1
        user: asa
```

## **🌊 컨테이너 안에서 작동하는 명령 지정 (command/entrypoint)**

- 컨테이너에서 실행될 명령은 `command`를 사용하여 지정

- 베이스 이미지에서 설정된 명령이 있을 경우 이를 덮어씀

```yaml
command: /bin/bash
```

  

`entrypoint`를 덮어쓸 수도 있음

```yaml
entrypoint:
  - php
  - -d
  - memory_limit=-1
```

## **🌊 컨테이너 간 네트워크 설정 (networks)**

- **🌊** ~~**컨테이너 간 연결 (links)**~~
    
    - 다른 컨테이너와 연결하려면 `links`를 사용하여 연결할 컨테이너명을 설정
    
    - 컨테이너명과 별도로 앨리어스명을 붙이고 싶을 때는 `서비스명:앨리어스명` 형태로 지정
    
    - 컨테이너 간 의존 관계를 설정하려면 `depends_on`을 사용
        
        - 서비스 시작 순서를 지정 가능
        
    
    ```yaml
    links:
      - logserver
      - logserver:log01  # 별칭 log01으로 연결
    ```
    
      
    
      
    
      
    

- 과거에는 `links`를 사용해 컨테이너를 연결했으나, 최신 Compose는 자동으로 같은 네트워크의 컨테이너를 연결
    
    ⇒ `links` 불필요해짐
    

- 같은 `docker-compose.yml` 내의 컨테이너는 자동으로 같은 네트워크를 사용하므로 links 없이도 통신 가능

- 서로 다른 네트워크를 사용하고 싶다면 `networks`를 사용
    
    - ==이 경우 컨테이너간 직접 연결이 불가능==
    

```yaml
services:
  app:
    image: my-app
    networks:
      - frontend
      - backend

  db:
    image: mysql
    networks:
      - backend

networks:
  frontend:
  backend: # backend 네트워크는 app과 db 모두 사용하므로,
           # backend 네트워크에서는 app과 db가 서로 통신할 수 있음!
```

## **🌊 컨테이너 간 통신 (ports/expose)**

### 📌 `ports`

- 컨테이너에서 공개할 포트는 `ports` 속성을 사용하여 지정
    
    - `호스트 머신의 포트 번호:컨테이너의 포트 번호`
    
    - 컨테이너의 포트만 지정
        
        - 이 경우, 호스트 머신의 포트는 랜덤한 값으로 설정됨
        
    

- YAML에서 `xx:yy` 형식은 시간으로 해석하므로, 반드시 문자열(`""`)로 정의

```yaml
ports:
  - "3000"
  - "8000:8000"
  - "49100:22"
  - "127.0.0.1:8001:8001"
```

  

### 📌 `expose`

- 호스트 머신의 포트를 공개하지 않고, **다른 컨테이너와의 연결을 위한 포트만 지정**(link 기능)
    
    - 웹 애플리케이션 서버를 경유하여 접근해야 하는 내부 서비스(로그 서버 등)에서 활용
    

```yaml
expose:
  - "3000"
  - "8000"
```

## **🌊 서비스의 의존관계 정의 (depends_on)**

- 여러 서비스의 의존관계를 정의할 땐 `depends_on`을 지정

- 컨테이너의 **시작 순서만 제어**할 뿐, 애플리케이션이 실제로 준비될 때까지 기다려주지는 않음
    
    - 데이터베이스 서비스가 완전히 준비되기 전에 웹 애플리케이션이 실행될 수 있으므로, 애플리케이션 측에서 대책을 세워야 함
        
        - `healthcheck` 와 함께 사용
            
            ```yaml
            services:
              web:
                image: nginx
                depends_on:
                  db:
                    condition: service_healthy
                    # db가 완전히 준비된 후에 실행
            
              db:
                image: mysql
                healthcheck: # db 준비 여부 확인
                  test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
                  interval: 10s
                  retries: 5
                  start_period: 30s
                  timeout: 5s
            ```
            
        
    

```yaml
services:
  webserver:
    build: .
    depends_on: # webserver가 db와 redis 컨테이너에 각각 의존
      - redis
      - db
  redis:
    image: redis
  db:
    image: postgres
```

## **🌊 컨테이너 환경변수 지정 (environment/env_file)**

### 📌 `environment`

- 컨테이너 내부에서 사용할 환경변수는 `environment`를 사용하여 지정

- 배열 형식 또는 해시 형식 중 하나로 변수 지정

```yaml
# 배열 형식으로 지정
environment:
  - HOGE=fuga
  - FOO=bar

# 해시 형식으로 지정
environment:
  HOGE: fuga
  FOO: bar
```

  

### 📌 `env_file`

- 설정해야 할 환경변수의 개수가 많을 경우, 별도의 환경변수 파일을 작성하여 읽어들일 수 있음

  

1️⃣ `envfile` 내용

```yaml
HOGE=fuga
FOO=bar
```

  

**2️⃣** `docker-compose.yml`에서 환경 변수 읽어들이기

```yaml
env_file: envfile
```

- 여러 개의 환경변수 파일을 불러올 수도 있으며, 배열 형태로 작성

- 파일 경로는 상대 경로 또는 절대 경로 모두 사용 가능

```yaml
env_file:
  - ./envfile1
  - /app/envfile2
  - /tmp/envfile3
```

> 
> 애플리케이션 내에서 사용하는 비밀정보(API 키 등)의 관리는 **컨테이너 오케스트레이션 툴**의 기능을 사용하는 게 좋음_==→ 9.4장에서 안내==_

## **🌊 컨테이너 정보 설정 (container_name/labels)**

### 📌 `container_name`

- Docker Compose로 생성되는 컨테이너에 이름을 지정하려면 `container_name`을 지정

- 컨테이너명은 고유해야 하므로, 커스텀명을 지정하면 여러 컨테이너로 스케일할 수 없음

```yaml
container_name: web-container
```

  

### 📌 `**labels**`

- 컨테이너에 라벨을 설정하려면 `labels`를 지정
    
    - 배열 형식 또는 해시 형식으로 지정
    

```yaml
# 배열 형식으로 지정
labels:
  - "com.example.description=Accounting webapp"
  - "com.example.department=Finance"

# 해시 형식으로 지정
labels:
  com.example.description: "Accounting webapp"
  com.example.department: "Finance"
```

설정된 라벨을 확인하려면 `docker-compose config` 명령어를 사용

## **🌊 컨테이너 데이터 관리 (volumes/volumes_from)**

### 📌 `volumes`

- 컨테이너에 볼륨을 마운트하려면 `volumes`를 지정

- 호스트 측 마운트 경로를 지정하려면 `호스트 디렉토리 경로:컨테이너 디렉토리 경로` 형식으로 지정

- 읽기 전용으로 마운트하려면 `:ro`를 붙임

```yaml
volumes:
  - /var/lib/mysql
  - cache/:/tmp/cache
  - ~/configs/:/etc/configs/:ro # 읽기 전용으로 마운트
```

  

### 📌 `volumes_from` (비권장)

- 다른 컨테이너의 모든 볼륨을 마운트하려면 `volumes_from`으로 컨테이너명을 지정
    
    - 다른 컨테이너의 볼륨을 **공유 받아** 사용
        
        → ‼명확성이 떨어지고 유지보수가 어려워 최신 Compose에서는 잘 사용하지 않음!
        
    

```yaml
volumes_from:
  - log # log라는 이름의 컨테이너의 모든 볼륨을 마운트
```

  

> [!important]
> 
> - `**volumes_from**`**과** `**volumes**` **차이점**
>     
>     |옵션|기능|
>     |---|---|
>     |`volumes`|**호스트 디렉토리 또는 Docker 볼륨을 직접 마운트**|
>     |`volumes_from`|**다른 컨테이너에서 마운트된 볼륨을 공유**|
>     
>     즉, `**volumes**`는 개별적으로 볼륨을 정의하고 마운트하는 방식이고, `**volumes_from**`은 기존 컨테이너의 볼륨을 그대로 가져와서 사용하는 방식이다.
>     
>       
>     
>     `volumes_from`은 소스 컨테이너가 삭제되면 연결된 볼륨도 사라질 수 있어 `volumes`를 사용하는 것이 더 안전하다.
>

### Docker Compose 기본 명령

|서브 명령|설명||서브 명령|설명|
|---|---|---|---|---|
|up|컨테이너 생성/시작||pause|컨테이너 일시 정지|
|ps|컨테이너 목록 표시||unpause|컨테이너 재개|
|logs|컨테이너 로그 출력||port|공개 포트 번호 표시|
|run|컨테이너 실행||config|구성 확인|
|start|컨테이너 시작||kill|실행 중인 컨테이너 강제 정지|
|stop|컨테이너 정지||rm|컨테이너 삭제|
|restart|컨테이너 재실행||down|리소스 삭제|

- docker-compose 명령은 docker-compose.yml을 저장한 디렉토리에서 실행되며, 다른 디렉토리의 docker-compose.yml 파일을 지정하려면 -f 옵션을 사용한다.

```docker
docker-compose -f ./~~~/~~/docker-compose.yml up
```

- 서브 명령 이후 특정 컨테이너 식별자를 사용해 해당 컨테이너만을 조정할 수 있다.

  

### 여러 컨테이너 생성 ( up )

- 작성한 docker-compose.yml 파일을 바탕으로 여러 개의 컨테이너를 생성/시작할 때는

```docker
docker-compose up
```

명령을 사용하며, 아래와 같은 옵션을 사용할 수 있다

- -d : 백그라운드 실행

- —no-deps : 링크 서비스를 시작하지 않는다.

- —build : 이미지를 빌드한다.

- —no-build : 이미지를 빌드하지 않는다.

- -t, —timeout : 컨테이너의 타임아웃을 초로 지정(기본 10초) 한다.

- —scale SERVICE=서비스 수 : 서비스 수를 지정한다.

### 컨테이너에서 명령 실행 ( run )

- Docker Compose로 시작한 컨테이너에서 임의의 명령을 실행하고 싶을 때는

```docker
docker-compose run
```

명령을 사용한다. ( up : 컨테이너를 생성, 실행 / run : 컨테이너 실행 )

  

### 여러 컨테이너 시작(start)/정지(stop)/재시작(restart)

- Docker Compose를 사용해 여러 개의 서비스를 일괄적으로 시작/일시 정지/정지/재시작 할 수 있다.

```docker
docker-compose start | stop | restart <컨테이너 식별자>
```

  

### 여러 컨테이너 일시 정지(pause)/재개(unpause)

- Docker Compose를 사용해 여러 개의 서비스를 일괄적으로 일시 정지/재개할 수 있다.

```docker
docker-compose pause | unpause
```

  

### 서비스의 구성 확인 ( port/config )

- 서비스의 공개용 포트 확인

```docker
docker-compose port [옵션] <서비스명> <프라이빗 포트 번호>
```

```docker
docker-compose port webserver 80
```

- webserver라는 이름의 서비스의 80번 포트에 할당되어 있는 설정을 확인한다.

- 지정할 수 있는 옵션
    
    - —protocol=proto : 프로토콜. tcp 또는 udp
    
    - —index=index : 컨테이너의 인덱스 수
    

  

- Compose의 구성 확인

```docker
docker-compose config
```

![[image 23.png]]

- Compose 구성 파일의 내용을 확인할 수 있다.

  

### 여러 컨테이너 강제 정지(kill) / 삭제(rm)

- 실행 중인 컨테이너를 강제로 정지 시킬 때 kill 명령어를 사용하며, 이 명령은 컨테이너에 시그널을 송신한다.

- 시그널 : 프로세스 사이 연락을 주고받을 수 있도록 하는 장치로, Linux 커널에 내장되어 있다.  
    실행 중인 프로세스의 처리를 멈추고 다른 프로세스를 퍼리하고 싶은 경우나 프로세스를 강제 종료하고 싶을 때 사용한다.

### 여러 리소스의 일괄 삭제
