- 브라우저와 중간 서버(CDN, Reverse Proxy)가 이미 받은 데이터를 저장해두었다가 다시 요청이 오면 다운로드를 다시 실행하지 않고 받아놓았던 데이터를 빠르게 제공하는 기술

### 목적
- 속도 향상(빠른 로딩)
- 트래픽 감소(네트워크 낭비 절감)
- 서버 부하 감소


### 종류
- HTTP Caching 의 종류는 크게 두 가지로 나뉜다.
1. **브라우저 캐시 ( Client Side Cache )**
	- 사용자 PC/모바일 브라우저 내부에 저장 ( 가장 빠름, 메모리/디스크에서 바로 읽음 )
	- 브라우저가 판단하는 캐시 저장 위치 ( 메모리 캐시 / 디스크 캐시 )

2. **중간 캐시 ( CDN / Reverse Proxy Cache )**
	- Cloudflare, Akamai, NGINX 등의 서버가 가지고 있는 캐시
	- 트래픽의 대부분을 여기서 처리

### 캐시 판단 기준
- HTTP header의 Cache-Control 값을 기준으로 파악한다.
- **Cache-Control 핵심 지시자**
	1. max-age : 얼마나 오래 캐싱해도 되는지 (초 단위)
	2. no-cache : 캐시 저장 가능, 하지만 재사용 하려면 서버에 확인하라. 브라우저는 원본이 바뀌었는지 서버세 체크 후 같으면 캐시 사용 ( 304 Modified )
	3. no-store : 캐싱 금지 ( 민감한 데이터 )
	4. must-revalidate : 캐시가 만료되면 반드시 서버에 검증 요청해야 한다
	5. public : CDN 포함 모든 캐시 허용
	6. private : 사용자 브라우저만 캐싱 가능
	7. immutable : 절대 변하지 않는 파일, 확인조차 하지 않음. 보통 JS/CSS 버전 해싱된 파일에 사용


### ETag와 Last-Modified (검증 캐싱)
- 캐시 만료 전이라면 아무 검증도 없이 바로 사용되지만, 만료 혹은 no-cache 인 경우 서버에 확인해야 하는데, 이 때 사용되는 것이 ETag와 Last-Modified 이다.
- **ETag ( 파일 fingerprint )**
	- 서버가 파일의 고유 해시값을 생성해 응답하고, 브라우저는 다음 요청에 해당 해시값을 보내면 서버가 같다고 판단 -> 304 Not Modified 응답 -> 파일 본문(body) 없음 -> 캐시 사용
- **Last-Modified ( 파일 수정 시간 )**
	- 응답의 파일 수정 시간과 브라우저 요청의 파일 수정 시간을 비교해 같으면 304, 다르면 새로운 파일 응답하는 방식


### 캐시의 3단계 구조
- HTTP 캐시는 **브라우저 메모리 -> 브라우저 디스크 -> CDN 엣지 -> CDN 중간 >- 오리진 서버** 중 가장 가까운 곳에서 해결된다.
- 가까운 곳에서 Hit(적중) 되면 속도가 극단적으로 빨라진다.


### Cache Hit / Cache Miss
- **Cache Hit**
	- 요청한 리소스가 캐시에 있음 -> 바로 반환
- **Cache Miss**
	- 캐시에 없음 -> 오리진에서 다운로드 후 저장


### 실무에서 가장 많이 쓰이는 캐싱 전략
1. 정적 파일(assets)은 장기 캐싱
	- JS / CSS / img / font 등은 보통 정적 파일에 해시를 붙여서 파일명이 바뀌면 새로운 파일로 간주함
2. HTML은 캐시가 매우 짧거나 아예 비활성화
	- HTML은 자주 바뀌기 때문
3. API 응답은 상황에 따라 다르게 캐싱
	- POST / PUT / DELETE 는 기본적으로 캐싱 금지


### 캐싱 전체 흐름 요약
1. 브라우저 요청
2. 캐시 확인
3. Hit -> 즉시 반환 / Miss -> CDN 확인 -> Hit(반환) / Miss(오리진 서버 요청)
4. 오리진 응답
5. 적절한 캐시 정책에 따라 저장
6. 사용자에게 반환

### \*CDN
- **Content Delivery Network**
- 웹 컨텐츠를 더 빠르고 안정적으로 전달하기 위한 전 세계 분산 서버 네트워크
- 사용자가 어디서 접속하든 가까운 위치에 있는 서버가 콘텐츠를 대신 제공해주는 시스템
- **CDN 사용 이유**
	- 모든 사용자가 원본 서버까지 직접 연결하기에는 응답 시간이 너무 기므로 가장 가까운 CDN 엣지 서버에서 응답해주면 속도가 빨라진다
	- 특정 지역에서의 지연 시간 증가 방지
	- CDN 엣지 서버에서 트래픽 분산 처리
	- 전 세계 배포 자동화
- **핵심 기능**
	- 캐싱 - 사용자가 요청한 파일을 서버가 아닌 CDN 엣지 노드에서 바로 제공하는 기능
	- 지연 감소
		한국에서 미국 서버까지 요청한다고 예를 들면, 시간이 오래 걸리기 때문에 CDN은 한국 또는 근처 region에 캐시를 보유해 지연을 최소화한다.
	- 트래픽 급증 보호
	- 전송 최적화(TFB / TTFB 개선)
		CDN은 GZip, Brotli 압축, HTTP/2, HTTP/3, TLS offload 등을 제공 -> 콘텐츠 전송 속도 향상
