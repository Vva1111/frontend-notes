## V8 엔진의 메모리 구조

![[Untitled 373.png]]

- Heap 메모리 : 객체 혹은 동적 데이터 저장 영역, 메모리 영역에서 가장 큰 블록이며, 가비지 컬렉션이 발생하는 곳이다. Heap 내부에서도 Young, Old 영역에서만 가비지 컬렉션이 실행된다.
    
    - New 영역(Young Generation) : 새로 만들어진 짧은 생명 주기를 가진 모든 객체를 저장
    
    - Old 영역 : ***마이너 GC**가 두 번 발생할 동안 New 영역에서 살아남는 객체들이 이동하는 영역. 이 영역은 ***메이저GC**가 관리하며, 두 개의 영역으로 나뉜다.
        
        - Old 포인터 영역 : 살아남은 객체들을 가지며, 이 객체들은 다른 객체를 참조한다.
        
        - Old 데이터 영역 : 데이터만 가진 객체들(다른 객체를 참조하지 않는다.) 을 가진다.
        
    
    - Large Object Space : 다른 영역의 제한된 크기보다 큰 객체들이 있는 영역. 해당 영역 개체들은 가비지 컬렉터로 이동하지 않는다.
    
    - 코드 영역 : 실시간(JIT) 컴파일러가 컴파일된 코드들을 저장하는 곳이다. 유일하게 실행 가능한 메모리가 있는 영역이다.(코드들은 ‘Large Object Space’에 할당될 수도 있고 실행도 가능하다.)
    

- Stack : V8 프로세스마다 하나의 스택을 가진다. 메서드와 함수 프레임, 원시 값, 객체 포인터를 포함한 정적 데이터가 저장되는 곳

  

> [!important] **마이너 GC(Garbage Collector)**
> 
> - New 영역(Young Generation)을 작고 깨끗하게 유지시킨다.
>     
>     객체들은 New 영역에 할당되고 크기가 매우 작다.
>     
> 
> - 마이너 CG에서 살아남은 객체들은 항상 새로운 곳으로 대피(evacuation) 한다.
> 
> - New 영역은 크기가 같은 2개의 세미영역(To, From)으로 나뉜다. 대부분의 할당은 To 영역에서 만들어진다. To 영역이 가득 차면 마이너 GC가 발생한다.
> 
> - 위의 대피 과정을 위해 언제나 1개의 세미 영역은 비어있다.(To space)
> 
> - 객체들이 머물러 있는 영역(From space)에서 To space로 이동할 때 살아남은 객체들은 연속적인(붙어있는) 메모리로 이동한다. 이는 메모리 단편화를 주기적으로 방지하는 장점이 있다.
> 
> - 객체는 새로운 주소값으로 포인터가 갱신된다.
> 
> ![[Untitled 374.png]]
> 
> - 객체 대피가 완료되면 From space에 남아있는 더 이상 쓸모 없는 객체들을 버린다.
> 
> - 마지막으로 To space와 From space의 역할을 바꿔준다.
> 
> ![[Untitled 375.png]]
> 
> - 여기서 새로운 객체가 할당되면, From space의 다음 빈 주소에 할당되고, 새로운 객체는 생존해 To space로 대피한다. 하지만 기존에 생존했던 4개의 객체는 한번 더 생존하면 To space가 아닌, Old space로 이동된다.

  

> [!important] **메이저 GC(Garbage Collector)**
> 
> - Old space 내부에 있는 객체를 정리하는 가비지 컬렉터
> 
> - Mark-Sweep-Compact, Tri-color 알고리즘 사용
>     
>     가비지 컬렉션 대상을 알기 위해 마팅해나가며 Tri-color(white, gray, black로 마킹한다.)
>     
>     white : GC가 아직 탐색하지 못한 상태
>     
>     Gray : 탐색은 했으나, 해당 객체가 참조하고 있는 객체가 있는지 확인하지 못한 상태
>     
>     Black : 해당 객체가 참조하고 있는 객체까지 확인 한 상태
>     

  

## 메모리 누수

- 프로그램이 사용 가능한 것보다 더 많은 메모리를 Heap 에 할당하려 할 때 메모리 부족 오류가 발생한다. Heap이 잘못 관리되면 메모리 누수가 발생할 수 있다.

- 부주의 또는 일부 프로그램 오류로 인해 더 이상 사용되지 않는 메모리를 찾아 해제하지 못하는 경우도 있다.

  

## 자바스크립트의 가비지 컬렉션

- 참조되지 않는( 사용되고 있지 않은 ) 메모리는 가비지 변수 또는 데이터가 된다. 만약 이런 데이터가 메모리에 계속 쌓인다면 결국에는 메모리 사용량을 초과하게 될 것이기 때문에 쓸모 없는 데이터는 정리해야 한다.

- 가비지 컬렉션 메커니즘은 자동과 수동 방식으로 나뉜다.

- C, C++는 수동 정리, 즉 개발자가 의도한 메모리 할당과 해제를 할 수 있다. 하지만 자바스크립트는 자동 정리 메커니즘은 가지고 있어 개발자는 메모리 할당과 해제에 관여할 수 없다.

- 이는 개발자는 메모리 관리에 대해 신경 쓸 필요가 없다는 뜻이 되기도 하지만, 다른 한편으로는 메모리 관리에 더욱 신경 써야 한다는 뜻이 될 수도 있다.

### 자바스크립트 가비지 컬렉션 기준

- 자바스크립트는 ‘도달 가능성(reachability)’ 이라는 개념을 사용해 메모리 관리를 수행한다.
    
    쉽게 말해, ‘어떻게든 접근하거나 사용할 수 있는 값’ 을 말한다. 이러한 값은 메모리에서 삭제되지 않는다.

    - 현재 함수의 지역 변수와 매개 변수
    
    - 중첩 함수의 체인에 있는 함수에서 사용되는 변수와 매개 변수
    
    - 전역 변수 등 root 라고 부르는 값들
    

- 루트 값이 참조하는 값이나 체이닝으로 루트에서 참조할 수 있는 값들은 도달 가능한 값이 된다.

### 가비지 컬렉션 내부 알고리즘

- mark-and-sweep 알고리즘

- 가비지 컬렉션은 대게 아래와 같은 단계를 거쳐 수행된다.
    
    1. 가비지 컬렉터는 루트 정보를 수집하고 이를 ‘mark(기억)’ 한다.
    
    1. 루트가 참조하고 있는 모든 객체를 방문하고 이것들을 ‘mark(기억)’ 한다.
    
    1. mark 된 모든 객체에 방문하고 그 객체들이 참조하는 객체도 mark 한다. 한 번 방문한 객체는 전부 mark 하기 때문에 같은 객체를 다시 방문하는 일은 없다.
    
    1. 루트에서 도달 가능한 모든 객체를 방문할 때 까지 위 과정을 반복한다.
    
    1. mark 되지 않은 모든 객체를 메모리에서 제거한다.

### 최적화 기법

- 자바스크립트 엔진은 실행에 영향을 미치지 않으면서 가비지 컬렉션을 더 빠르게 하는 다양한 최적화 기법을 적용한다.
    
    1. **generational collection(세대 별 수집)**
        
        객체를 ‘새로운 객체’와 ‘오래된 객체’로 나눈다. 객체 상당수는 생성 이후 제 역할을 빠르게 수행해 금방 쓸모가 없어지는데, 이런 객체를 ‘새로운 객체’로 구분한다.
        
        가비지 컬렉터는 이런 객체를 공격적으로 메모리에서 제거한다.
        
        일정 시간 이상 동안 살아남은 객체는 ‘오래된 객체’로 분류하고, 가비지 컬렉터가 덜 감시한다.
        
    
    1. **incremental collection(점진적 수집)**
        
        방문해야 할 객체가 많다면, 모든 객체를 한번에 방문하고 mark 하는데 상당한 시간이 소모된다. 이렇게 되면 가비지 컬렉션에 많은 리소스가 사용되어 실행 속도도 느려질 것이다.
        
        자바스크립트 엔진은 이런 현상을 개선하기 위해 가비지 컬렉션을 여러 부분으로 분리한 다음, 각 부분을 별도로 수행한다.
        
        작업을 분리하고 변경 사항을 추적하는데 추가 작업이 필요하긴 하지만 긴 지연을 짧은 지연 여러 개로 분산시킬 수 있다는 장점이 있다.
        
    
    1. **Orinico ( V8 )**
        
        - 마이너 / 메이저 GC를 실행할 때 프로그램이 멈추는데, 이를 stop-the-world 라고 한다. 이 시간이 길어질수록 페이지가 느려지거나 렌더링이 지연되어 사용자들의 UX 경험이 나빠지게 된다.
        
        - Orinocno 프로젝트는 이런 문제를 해결하기 위해 아래와 같은 기술들을 추가했다.
        
        - Parallel
            
            기존에는 메인 쓰레드 혼자 하던 일을 헬퍼 쓰레드들과 균등히 나눠 작업한다. 쓰레드 간의 동기화를 처리해야 해서 오버헤드는 생기지만, stop-the-world 시간이 크게 감소가한다.
            
            ![[Untitled 385.png]]
            
        
        - Incremental
            
            메인 쓰레드가 적은 양의 작업을 간헐적으로 처리한다. 메인 쓰레드에서 가비지 컬렉션에 소요하는 시간이 분산되어 더 좋은 UX 제공이 가능하다.
            
            ![[Untitled 386.png]]
            
        
        - Concurrent
            
            메인 쓰레드는 더 이상 가비지 컬렉션을 하지 않고, 헬퍼 쓰레드들이 수행한다. 기술적으로 구현하기는 어렵지만 메인 쓰레드의 stop-the-world 시간이 전혀 없다는 큰 장점이 있다.
            
            ![[Untitled 387.png]]
            
        
        - Idle-time GC
            
            개발자는 GC에 직접 접근할 수 없지만, V8은 크롬과 같은 embedder에게 가비지 컬렉션을 유발할 수 있는 메커니즘을 제공한다. 크롬은 프로그램이 쉬는 free나 idle time을 알 수 있다. 예를 들면, 1초에 60프레임을 제공하는 크롬은 1프레임을 렌더링 하기 위해 약 16ms가 소모되는데, 만약 애니메이션 프레임 렌더링 작업이 16ms보다 빨리 끝나면 크롬은 다음 프레임 작업 전까지 가비지 컬렉션을 유발한다.

### 자바스크립트의 가비지 컬렉션 메커니즘(실행 컨텍스트 관점)

- 일반적으로 전역 변수는 자동으로 정리되지 않는다.

- 가비지 컬렉션은 로컬 스코프 메모리 수집에 초점이 맞춰져 있다.

```jsx
function fn1 (){
	let a = {
		name: 'byteFish'
	};

	let b = 3;

	function fn2() {
		let c = [1, 2, 3];
	}

	fn2();
	
	return a;
}
let res = fn1();
```

- 위 코드의 호출 스택은 아래와 같다.

![[Untitled 389.png]]

- 그림의 왼쪽은 Stack 영역으로 실행 컨텍스트와 원시 타입의 데이터를 저장하는데 사용되고, 오른쪽은 Heap 영역으로, 객체를 저장하는데 사용된다.

- fn2() 가 실행될 때, 콜 스택 내부 실행 컨텍스트는 위에서부터 fn2 → fn1 → 전역 순으로 존재한다.

- fn2() 가 실행을 완료하면, fn2() 는 실행 컨텍스트에서 제거되고, 메모리 공간은 해제된다.

- 자바스크립트 가비지 컬렉션은 가끔 호출 스택을 탐색하고 사용되지 않는 메모리를 수집한다.

- 위 시점에서 가비지 컬렉션이 수행된다고 가정하면, 가비지 수집기가 호출 스택을 순회할 때 변수 b와 c가 사용되지 않음을 발견해 가비지 데이터임을 확인하고 표시한다.

- fn1() 함수는 실행 후 변수 a를 반환하고 전역 변수 res에 할당되므로 활성 데이터로 식별되고 그에 따라 표시한다.

- 유휴 시간에 가비지 데이터로 표시된 모든 변수는 아래 그림과 같이 해당 메모리를 해제하기 위해 지워진다.

## 메모리 누수의 예

### 1. 클로저의 잘못된 사용

- 상단 예제에서 fn1() 함수 실행 컨텍스트가 종료된 후, 해당 스코프의 실행 컨텍스트 내부에 있던 변수 a는 가비지 데이터로 제거되었어야 했지만 fn1() 함수가 내부 변수 a를 반환하고 전역 변수 res에 할당됨으로써 res는 변수 a의 값을 참조하게 되었다.

- 그렇기 때문에 변수 a는 사용중이라는 표식을 가지게 되고, 메모리를 차지할 수 있게 된다.

- 하지만 추후에 전역 변수 res를 사용하지 않는다면 해당 방식은 메모리 누수를 일으키는 원인이 된다.
    
    (res 전역 변수를 계속 사용한다면 상관 없다.)
    
      
    

### 2. 의도치 않게 생성된 전역 변수

- 전역 변수는 가비지 컬렉터에 의해 수집되지 않는다. 그렇기 때문에 전역 변수 선언은 가능한 적게 해야한다.

- DOM 노드를 직접 제거했을 때, DOM 노드의 메모리를 해제했어야 하지만 몇몇 코드는 여전히 삭제된 노드를 참조하고 있다.

### 4. 콘솔 출력

- 콘솔 출력 또한 메모리 누수의 원인이 될 수 있다.

- 콘솔 탭을 열기 위해서는 브라우저가 객체의 정보를 저장하고 가지고 있어야 해당 객체의 정보를 출력할 수 있다.

- 개발 환경에서 사용한 console 객체는 실제 배포 전에 모두 삭제하는 것이 권장된다.

  

### 5. 해제하지 않은 타이머

- 해제하지 않은 타이머도 메모리 누수의 원인이 될 수 있다.

- 타이머 함수가 실행하고자 하는 콜백 함수를 참조하고 있기 때문에 해당 함수의 메모리는 여전히 사용중인 메모리로 잡히게 된다.

- 따라서 지속적인 타이머의 사용이 필요하지 않다면 타이머 작동 횟수를 정해 놓고 정해진 횟수 이후로는 타이머가 정지하게 설정 해 놓는 것도 메모리 누수를 막을 수 있는 좋은 방법이다.

  

> [!important] 자바스크립트 가비지 컬렉션이 자동으로 이루어지지만 특정 변수들의 메모리를 수동으로 해제하는 일 또한 메모리 누수를 막는데 큰 도움이 된다. 예를 들어, 더 필요하지 않은 변수가 외부 변수에 의해 참조되고 있어 메모리가 해제될 수 없을 때 null을 할당해서 다음 가비지 컬렉션이 동작할 때 메모리를 해제할 수 있도록 도와주는 것이다.
