### Item 45. devDevendencies에 typescript와 @types 추가하기

- npm은 세 가지 종류의 의존성을 구분해 관리하며, 각각의 의존성은 package.json 파일 내의 별도 영역에 들어있다.

- dependencies
    
    - 프로젝트를 실행하는데 필수적인 라이브러리들을 포함
    
    - 프로젝트를 npm에 공개해 다른 사용자가 해당 프로젝트를 설치한다면 dependencies에 들어 있는 라이브러리도 함께 설치된다.
    

- devDependencies
    
    - 프로젝트 개발, 태스트 하는데 사용되지만, 런타임에는 필요 없는 라이브러리들이 포함된다.
    
    - npm에 프로젝트를 공개해 다른 사용자가 설치한다면 devDependencies의 라이브러리들은 제외된다.
    

- peerDependencies
    
    - 런타임에 필요하긴 하지만, 의존성을 직접 관리하지 않는 라이브러리들이 포함된다.
    
    - 단적인 예로, 플러그인을 들 수 있다.
    

- 모든 타입스크립트 프로젝트에서 공통적으로 고려해야 할 의존성 두가지
    
    - 타입스크립트 자체 의존성을 고려해야 한다.
        
        - 타입스크립트를 시스템 레벨로 설치하기 보다는 devDependencies에 넣는 것이 좋다.
        
        - devDependencies에 포함되어 있다면, npm install 로 모든 팀원들이 동일한 타입스크립트 버전을 설치할 수 있다.
        
        - 또한 버전 업데이트는 다른 라이브러리들의 업데이트와 같은 방식을 사용하게 된다.
        
    
    - 타입 의존성(@types)을 고려해야 한다.
        
        - 사용하려는 라이브러리에 타입 선언이 포함되어 있지 않다 해도DefinitelyTyped 에서 타입 정보를 얻을 수 있다.
        
    

  

  

  

### Item 46. 타입 선언과 관련된 세 가지 버전 이해하기

- 타입스크립트를 사용하면서 고려해야 할 세가지 사항
    
    - 라이브러리의 버전
    
    - 타입 선언의 버전
    
    - 타입스크립트의 버전
    

- 라이브러리 관리의 메커니즘을 이해하게 된다면, 프로젝트 내에서 작성한 타입 선언을 외부에 공개해야 하는 시점이 되었을 때, 버전과 관련해서 제대로 된 결정을 내릴 수 있다.

- 실제 라이브러리와 타입 정보의 버전이 별도로 관리되는 방식은 다음의 네 가지 문제점이 있다.
    
    - 라이브러리를 업데이트 했지만 실수로 타입 선언은 업데이트 하지 않은 경우, 만약 타입 선언이 업데이트에 맞춰지지 않았다면 보강 기법을 사용하거나 타입 선언 업데이트를 직접 작성해 라이브러리에 기여하는 방법이 있다.
    
    - 라이브러리보다 타입 선언의 버전이 최신인 경우, 타입 체커는 최신 API 기준으로 코드를 검사하지만 런타임에 실제로 쓰이는 것은 과거 버전이기 때문에 두 가지 버전을 맞춰줘야 한다.
    
    - 프로젝트와 라이브러리가 서로 필요로 하는 타입스크립트 버전이 다를 경우
    
    - @types 의존성이 중복되는 경우
    

- 번들된 타입에서는 @types 의 버전 선택이 불가능

- 타입스크립트로 작성된 라이브러리의 경우, 라이브러리에 타입 선언을 포함하는 방식은 잘 작동한다. 하지만, 자바스크립트로 작성된 라이브러리라면 타입 선언에 오류가 많을 수 있고, 잦은 업데이트를 필요로 하기 때문에 DefinitelyTyped 에 공개해 커뮤니티에서 관리하는 방식이 더 유리하다

  

  

  

### Item 47. 공개 API에 등장하는 모든 타입을 익스포트하기

- 라이브러리 제작자는 프로젝트 초기에 타입 익스포트부터 작성해야 한다.

- 만약 함수의 선언에 이미 타입 정보가 있다면 제대로 익스포트 되고 있는 것이며, 타입 정보가 없다면 타입을 명시적으로 작성해야 한다.

- 공개 api 매개변수에 놓이는 순간 타입은 노출된다.

  

  

  

### Item 48. API 주석에 TSDoc 사용하기

- 주석은 JSDoc 스타일로 만드는 것이 좋다

- 대부분의 편집기는 함수가 호출되는 곳에서 함수에 붙어 잇는 JSDoc 스타일의 주석을 툴팁으로 표현해주기 때문이다.

- 그러나 인라인 주석은 편집기가 표시해주지 않는다.

- 타입스크립트 언어 서비스가 JSDoc 스타일을 지원한다.

- JSDoc 에는 @param, @returns 등과 같은 일반적인 규칙을 사용할 수 있다.

- 타입 정의에 TSDoc을 사용할 수도 있다.

- JSDoc에는 타입 정보를 명시하는 규칙(@param { string } name …)이 있지만, 타입스크립트에서는 타입 정보가 코드에 있기 때문에 TSDoc에서는 타입 정보를 명시하면 안된다.

  

  

  

### Item 49. 콜백에서 this에 대한 타입 제공하기

- this는 렉시컬 스코프인 변수들과는 다르게 다이나믹 스코프 방식이다. 이는 호출된 방식에 따라 스코프가 달라지는 것을 뜻한다.

- this는 전형적으로 객체의 현재 인스턴스를 참조하는 클래스에서 가장 많이 사용된다.

- 자바스크립트에서 this를 call()를 사용해 온전히 컨트롤 할 수 있다.

- DOM 에서도 this를 바인딩 할 수 있다.

- this 바인딩은 콜백함수에서 종종 사용된다.

- 화살표 함수를 사용하면 class가 생성될 때 마다 제대로 바인딩 된 this를 가지는 새 함수를 생성한다.

- this 바인딩은 자바스크립트의 동작이기 때문에, 타입스크립트 역시 this 바인딩을 그대로 모델링 하게 된다.

- 콜백 함수의 매개변수에 this 를 추가하면 this 바인딩이 체크되기 때문에 실수를 방지할 수 있다. 또한, 라이브러리 사용자의 콜백 함수에서 this를 참조할 수 있고, 완전한 타입 안전성도 얻을 수 있다.

- 콜백 함수에서 this 값을 사용해야 한다면 this는 API의 일부가 되는 것이기 때문에 반드시 타입 선언에 포함해야 한다.

  

  

  

### Item 50. 오버로딩 타입보다는 조건부 타입 사용하기

- 타입스크립트에서 함수의 구현체는 하나지만, 타입은 여러 개가 될 수 있다. 이를 활용해 함수의 타입을 개선할 수 있다.

- 타입스크립트는 오버로딩 타입 중에서 일치하는 타입을 찾을 때까지 순차적으로 검색한다.

- 오버로딩 타입을 추가해 문제를 해결할 수도 있지만, 조건부 타입을 사용하는 방법이 더 적절하다

- 조건부 타입은 타입 공간의 if 구문과 같다

- 조건부 타입은 삼항연산자 처럼 사용하면 된다.

- 조건부 타입은 개별 타입의 유니온으로 일반화하기 때문에 타입이 더 정확해진다.

- 각각의 오버로딩 타입이 독립적으로 처리되는 반면, 조건부 타입은 타입 체커가 단일 표현식으로 받아들이기 때문에 유니온 문제를 해결할 수 있다.

  

  

  

### Item 51. 의존성 분리를 위해 미러 타입 사용하기

- 만약 작성하는 라이브러리가 의존하는 라이브러리의 구현과 무관하게 타입에만 의존한다면, 필요한 선언부만 추출해 작성 중인 라이브러리에 넣는 것(미러링)을 고려해 보는 것도 좋다.

- 다른 라이브러리의 타입이 아닌 구현에 의존하는 경우에도 동일한 기법을 적용할 수 있고 타입 의존성을 피할 수 있다.
    
    - 그러나 프로젝트의 의존성이 다양해지고 필수 의존성이 추가됨에 따라 미러링 기법을 적용하기 어려울 수 있다.
    
    - 다른 라이브러리의 타입 선언의 대부분을 추출해야 한다면, 차라리 명시적으로 @types 의존성을 추가하는게 낫다.
    

- 필수가 아닌 의존성을 분리할 때는 구조적 타이핑을 사용하면 된다.

- 공개한 라이브러리를 사용하는 자바스크립트 사용자가 @types 의존성을 가지지 않게 해야 한다.

- 웹 개발자가 NodeJS 관련 의존성을 가지지 않게 해야 한다.

  

  

  

  

### Item 52. 테스팅 타입의 함정에 주의하기

- 타입 선언도 테스트를 거쳐야 한다.

- 그러나 타입 선언을 테스트하기는 매우 어렵기 때문에, 궁극적으로는 dtsLint 또는 타입 시스템 외부에서 타입을 검사하는 유사한 도구를 사용하는 것이 더 안전하고 간단하다.

- 타입 시스템 내에서 암시적 any 타입을 발견해 내는 것은 너무 어렵기 때문에 타입 체커와 독립적으로 동작하는 도구를 사용해 타입 선언을 테스트하는 방법이 권장된다.

- DefinitelyTyped의 타입 선언을 위한 도구는 dtsLint이다.

- 타입을 테스트할 때는 특히 함수 타입의 동일성(equality)과 할당 가능성(assignability)의 차이점을 알고 있어야 한다.

- 콜백이 있는 함수를 테스트할 때, 콜백 매개변수의 추론된 타입을 체크해야 한다. 또한, this가 API의 일부분이라면 역시 테스트 해야 한다.
