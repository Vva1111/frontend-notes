# CSS 이벤트

## replace ( oldClassName, newClassName )

- replace 메서드는 class 어트리뷰트에서 첫 번째 인수로 전달한 문자열을 두 번째 인수로 전달한 문자열로 변경한다.

```jsx
$box.calssList.replace('red', 'blue'); // class="box red" -> class="box blue"
```

## toggle(className[, force])

- toggle 메서드는 class 어트리뷰트에 인수로 전달한 문자열과 일치하는 클래스가 존재하면 제거하고, 존재하지 않으면 생성한다.

```jsx
$box.classList.toggle('foo'); // class="box blue" -> class="box blue foo"
or
$box.classList.toggle('foo'); // class="box blue foo" -> class="box blue"
```

## window.getComputedStyle(element[, pseudo])

- 요소에 적용되어 있는 CSS 스타일 참조

- style 프로퍼티는 인라인 스타일만 반환한다. 따라서 클래스를 적용한 스타일이나 상속을 통해 암묵적으로 적용된 스타일은 .style 프로퍼티로 참조할 수 없다.

- HTML 요소에 적용되어 있는 모든 CSS 스타일을 참조해야 할 경우 getComputedStyle 메서드를 사용한다.

- 첫 번째 인수(element)로 전달한 요소 노드에 적용되어 있는 평가된 스타일을 CSSStyleDeclaration 객체에 담아 반환한다.
    
    - 평가된 스타일이란, 요소 노드에 적용되어 있는 **모든** 스타일, 즉 링크 스타일, 임베딩 스타일, 인라인 스타일, 자바스크립트에서 적용한 스타일, 상속된 스타일, 기본(user agent) 스타일 등 모든 스타일이 조합되어 최종적으로 적용된 스타일을 말한다.
    

  

# 값 변경 이벤트

## readystatechange

- HTML 문서의 로드와 파싱을 나타내는 document.readyState 프로퍼티 값으로 document의 로딩 상태를 파악할 수 있는 속성
    
    - **loading**
    
    - **interactive** : document의 로드가 완료되고 구문 분석은 완료 되었지만 서브 리소스(스트립트, 이미지, 스타일시트, 프레임)는 여전히 로딩중인 상태, DOMContentLoaded 이벤트가 곧 실행될 것임을 나타낸다.
    
    - **complete** : 로드가 완료 되었고, load 이벤트가 곧 실행된 것임을 나타낸다.
    
    ( **load event** : DOMContentLoaded와 대조되는 이벤트로, 리소스 로드가 완료될 때까지 기다리지 않고 페이지 DOM이 로드되자마자 실행한다. load 이벤트는 완전히 로드된 페이지를 감지할 때만 사용되어야 한다. )
    

  

# DOM 뮤테이션 이벤트

## DOMContentLoaded

- HTML 문서가 완전히 파싱되고 모든 지연된 스트립트가 다운로드 및 실행되면 발생한다. 이미지, 하위 프레임, 비동기 스크립트 등의 로드가 완료될 때까지 기다리지 않는다.

- 스타일시트의 로드를 기다리지 않지만, 지연된 스크립트는 스타일시트를 기다리며DOMContentLoaded 이벤트는 지연된 스크립트 이후에 대기열로 추가된다.

- 또한, 지연되지 않거나 비동기가 아닌 스크립트(예: <script> )는 이미 구문 분석된 스타일시트가 로드될 때까지 기다린다.

  

  

# 이벤트 핸들러 프로퍼티 방식과 addEventListner 메서드 방식

- 이벤트 핸들러 프로퍼티 방식과 addEventListener 메서드 방식 모두 이벤트 핸들러 내부의 this는 이벤트를 바인딩 한 DOM 요소를 가리킨다.

- 즉, 이벤트 핸들러 내부의 this는 이벤트 객체의 currentTarget 프로퍼티와 같다.

```jsx
const $button1 = document.querySelector('.btn1');

$button1.onclick = function (e) {
	// this는 이벤트를 바인딩한 DOM 요소를 가리킨다.
	console.log(this); // $button1
	console.log(e.currentTarget); // $button1
	console.log(this === e.currentTarget); // true
}
```

- 화살표 함수로 정의한 이벤트 핸들러 내부의 this는 상위 스코프의 this를 가리킨다. 화살표 함수는 함수 자체의 this 바인딩을 가지지 않는다.

```jsx
...
<button class="btn1"> 0 </button>
...
const $button1 = document.querySelector('.btn1');

$button.oncilck = e =>{
	// 화살표 함수 내부의 this는 상위 스코프의 this를 카리킨다.
	console.log(this); // window
	console.log(e.currentTarget); // $button1
	console.log(this == e.currentTarget); // false
}
```

# 커스텀 이벤트 객체

```jsx
// KeybooardEvent 생성자 함수로 keyup 이벤트 타입의 커스텀 이벤트 객체 생성
const keyboardEvent = new keyboardEvent('keyup');
console.log(keyboardEvent.type); // keyup

// CustomEvent 생성자 함수로 foo 이벤트 타입의 커스텀 이벤트 객체를 생성
const customEvent = new CustomEvent('foo');
console.log(customEvent.type); // foo

// MouseEvent 생성자 함수로 click 이벤트 타입의 커스텀 이벤트 객체 생성
const customEvent = new MouseEvent('click'');
console.log(customEvent.type); // click
console.log(customEvent.bubbles); // false
console.log(customEvent.cancelable); // false

// 커스텀 이벤트 객체의 bubbles, cancelable 프로퍼티를 true로 설정하려면
//  이벤트 생성자 함수의 두 번째 인수로 전달해야 한다.
const customEvent = new MouseEvent('click', {
	bubbles: true,
	cancelabel: true,
	cilentX: 50,
	cilentY: 100
});

// keyboardEvent 생성자 함수로 keyup 이벤트 타입의 커스텀 이벤트 객체 생성
const keyboardEvent = new KeyboardEvent('keyup', { key: 'Enter' });
console.log(keyboardEvent.key); // Enter
```

- 생성된 커스텀 이벤트 객체는 버블링되지 않으며 preventDefault 메서드로 취소할 수도 없다.

- 즉, 커스텀 이벤트 객체는 bubbles와 cancelable 프로퍼티의 값이 false로 기본 설정된다.

- 커스텀 이벤트 객체에는 bubbles 또는 cancelable 프로퍼티 뿐만 아니라 이벤트 타입에 따라 가지는 고유의 프로퍼티 값을 지정할 수 있다.
    
    - 예를 들어,
    
    - 마우스 이벤트에서는 마우스 포인터의 좌표 정보를 나타내는 마우스 이벤트 객체 고유의 프로퍼티 screenX/Y, clientX/Y, pageX/Y, offsetX/Y와 버튼 정보를 나타내는 프로퍼티 altKey, ctrlKey 등 을 가진다.
    
    - 이러한 이벤트 객체 고유의 프로퍼티 값을 지정하려면 위 예제와 같이 이벤트 생성자 함수의 두 번째 인수로 프로퍼티를 전달한다.
    

- 이벤트 생성자 함수로 생성한 커스텀 이벤트는 isTrusted 프로퍼티의 값이 언제나 false 이다.

- 커스텀 이벤트가 아닌 사용자의 행위에 의해 발생한 이벤트에 의해 생성된 이벤트 객체의 isTrusted 프로퍼티 값은 언제나 true이다.

  

# 자바스크립트 비동기 함수 동작 방식

## 자바스크립트 메모리 구조 / 실행 컨텍스트

- 비동기 처리에서 소스코드의 평가와 실행을 제외한 모든 처리는 자바스크립트 엔진을 구동하는 환경인 브라우저 또는 Node.js가 처리한다.

- **Call Stack**
    
    - 소스 코드(전역 혹은 지역 등) 평가 과정에서 생성된 실행 컨텍스트가 추가되고 제거되는 스택 자료구조 영역
    
    - 함수를 호출하면 함수 실행 컨텍스트가 순차적으로 콜 스택에 푸시되어 들어온 순서대로 실행된다.
    
    - 자바스크립트 엔진은 단 하나의 콜 스택을 사용하기 때문에 최상위 실행 컨텍스트(실행 중인 컨텍스트)가 종료되어 콜 스택에서 제거되기 전 까지는 다른 어떤 태스크도 실행되지 않는다.
    

  

- **Heap**
    
    - **객체**가 저장되는 메모리 공간이다. 콜 스택의 요소인 실행 컨텍스트는 힙에 저장된 객체를 참조한다.
    
    - 메모리에 값을 저장하려면 먼저 값을 저장할 메모리 공간의 크기를 결정해야 한다.
    
    - 객체는 원시 값과는 달리 크기가 정해져 있지 않으므로, 할당해야 할 메모리 공간의 크기를 런타임에 결정(동적 할당) 해야 한다.
    
    - 따라서 객체가 저장되는 메모리 공간인 힙은 구조화되어 있지 않다는 특징이 있다.
    

  

- 비동기 처리에서 소스코드의 평가와 실행을 제외한 모든 처리는 자바스크립트 엔진을 구동하는 환경인 브라우저 또는 Node.js가 담당한다.

- 비동기 방식으로 동작하는 setTimeout 의 콜백 함수의 평가와 실행은 자바스크립트 엔진이 담당하지만, 호출 스케줄링을 위한 타이머 설정과 콜백 함수의 등록은 브라우저 또는 Node.js가 담당한다. 이를 위해 브라우저 환경은 태스크 큐와 이벤트 루프를 제공한다.

- **Task Queue**
    
    - setTimeout 이나 setInterval 과 같은 비동기 함수의 콜백 함수 또는 이벤트 핸들러가 **일시적으로 보관**되는 영역.
    
    - 태스트 큐와는 별도로 프로미스의 후속 처리 메서드의 콜백 함수가 일시적으로 보관되는 마이크로태스크 큐도 존재한다.
    

  

- **Event Loop**
    
    - 이벤트 루프는 콜 스택에 현재 실행 중인 실행 컨텍스트가 있는지, 그리고 태스크 큐에 대기 중인 함수(콜백 함수, 이벤트 핸들러 등)가 있는지 반복해서 확인한다.
    
    - 콜 스택과 태스크 큐 모두 감시
    
    - 만약 콜 스택이 비어있고 태스크 큐에 대기 중인 함수가 있다면 이벤트 루프는 **순차적(First In First Out)**으로 태스크 큐에 대기 중인 함수를 콜 스택으로 이동시킨다.
    
    - 이 때, 콜 스택으로 이동한 함수는 실행된다. 즉, 태스크 큐에 일시 보관된 함수들은 비동기 처리 방식으로 동작한다.
    

  

## 비동기 실행 예제

```jsx
function foo() {
	console.log('foo');
}

function bar() {
	console.log('bar');
}

setTimeout(foo, 0);
bar();
```

1. 전역 코드가 평가되어 전역 실행 컨텍스트가 생성되고 콜 스택에 푸시된다.

1. 전역 코드가 실행되기 시작해 setTimeout 함수가 호출된다. 이때 setTimeout 함수의 함수 실행 컨텍스트가 생성되고 콜스택에 푸시되어 현재 실행 중인 실행 컨텍스트가 된다. 브라우저의 Web API(호스트 객체) 인 타이머 함수도 함수이므로 함수 실행 컨텍스트를 생성한다.

1. setTimeout 함수가 실행되면 콜백 함수를 호출 스케줄링 하고 종료되어 콜 스택에서 팝된다. 이 때, 호출 스케줄링, 즉 타이머 설정과 타이머가 만료되면 콜백 함수를 태스크 큐에 푸시하는 것은 브라우저의 역할이다.

1. 브라우저가 수행하는 a와 자바스크립트 엔진이 수행하는 b는 병행 처리 된다.
    
    1. 브라우저는 타이머를 설정하고 타이머의 만료를 기다린다. 이후 타이머가 만료되면 콜백 함수 foo가 태스크 큐에 푸시된다. 만약 위와 같이 타이머가 0으로 지정된 경우, 최소 지연 시간 4ms가 지정된다. 이 처리 또한 브라우저가 수행한다. 이처럼 setTimeout 함수로 호출 스케줄링한 콜백 함수는 정확히 지연 시간 후에 호출된다는 보장은 없다. 지연 시간 이후에 콜백 함수가 태스크 큐에 푸시되어 대기하게 되지만 콜 스택이 비어야 호출되므로 약간의 시간차가 발생할 수 있다.
    
    1. bar 함수가 호출되어 bar 함수의 함수 실행 컨텍스트가 생성되고 콜 스택에 푸시되어 현재 실행 중인 실행 컨텍스트가 된다. 이후 bar 함수가 종료되어

## 자바스크립트 엔진 싱글 스레드 / 브라우저 멀티 스레드 (GPT 인용)

- 자바스크립트 엔진은 **싱글 스레드**로, 브라우저 전체는 **멀티 스레드**로 동작한다.

- 브라우저는 렌더링, 이벤트 처리, 네트워크 등의 작업을 병렬로 처리할 수 있다

- 위 구조에서 **JS는 요청만** 하고, 백그라운드 스레드(Web APIs 등)가 작업을 처리한 후, 완료 되면 콜백 큐에 넘기는 방식으로 작업한다.

- 자바스크립트 엔진
    
    - 싱글 스레드
    
    - 코드 실행, 함수 호출, 콜백 처리, setTimeout, Promise.then 같은 것을 처리
    
    - 내부에는 **Call Stack, Task Queue, Microtask Queue** 가 있음
    

- 브라우저
    
    - **렌더링 엔진 ( Repaint / Reflow )** - HTML/CSS 처리 및 UI 업데이트
    
    - **Web APIs** - setTimeout, DOM 이벤트, fetch 등 처리
    
    - **이벤트 루프** - 콜백 큐를 모니터링해서 Call Stack이 비면 작업 할당
    
    - **네트워크 스레드** - fetch, XHP 등의 요청 처리
    
    - **타이머 스레드** - setTimeout, setInterval 처리
    
    - **Web Worker** - JS 백그라운드 스레드 실행 ( 명시적으로 생성해야 함 )
        
        - 브라우저가 별도 스레드에서 JS 실행을 지원하도록 해주는 기능
        
    
    ```jsx
    const worker = new Worker('workwr js');
    woorker.postMessage('작업 시작');
    // worker.js는 다른 스레드에서 실행
    ```
    

  

- 실행 순서

```jsx
console.log("1");

setTimeout(() => {
  console.log("2");
}, 1000);

console.log("3");
```

1. “1” → 즉시 실행

1. setTimeout → 브라우저의 타이머 스레드에게 위임

1. “3” → 계속 실행

1. 타이머 1초 뒤 완료 → 콜백이 Task Queue에 들어감

1. Call Stack이 비면 → Event Loop가 task Queue에서 콜백을 실행

1. “2” → 출력

- JS 코드 실행 → Web API 호출 ( 비동기 ) → 브라우저가 처리 → 완료되면 콜백 큐 푸시 → JS 엔진이 콜백 실행
