- 프로그램의 일부분에만 타입 시스템을 적용할 수 있다는 특성 덕분에 점진적인 마이그레이션이 가능하다.

- 마이그레이션 할 때, any를 현명하게 사용할 수 있어야 한다.

  

### Item 38. any 타입은 가능한 좁은 범위에서만 사용하기

- 의도하지 않은 타입 안전성의 손실을 피하기 위해 any의 사용 범위를 최소한으로 좁혀야 한다.

- 함수의 반환 타입이 any인 경우, 타입 안정성이 나빠진다. 따라서, any 타입을 반환하면 절대 안된다.

- 강제로 타입 오류를 제거하려면 any 대신 @ts-ignore 를 사용하는 것이 낫다

  

  

  

### Item 39. any를 구체적으로 변형해서 사용하기

- any는 자바스크립트에서 표현할 수 있는 모든 값을 아우르는 매우 큰 범위의 타입이다.

- 숫자, 문자 뿐만 아니라 함수, 클래스, DOM, null, undefined 도 포함한다.

- 일반적인 상황에서는 any보다 구체적으로 표현할 수 있는 타입이 존재할 가능성이 높기 때문에 더 구체적인 타입을 찾아 타입 안정성을 높일 수 있도록 해야한다.

- any 타입의 값을 그대로 정규식이나 함수에 넣는 것은 권장되지 않는다.

- 함수의 매개변수를 구체화할 때, (요소의 타입에 관계 없이) 배열의 배열 형태라면 **any[][]** 처럼 선언하면 된다.

- 함수의 매개변수가 객체이긴 하지만 값을 알 수 없다면 **{ [key: string]: any }** 처럼 사용하면 된다.

- 위 방법 대신, 모든 비 기본형 타입을 포함하는 object 타입을 사용할 수 도 있다.
    
    - object 타입은 객체의 키를 열거할 순 있지만 속성에 접근할 수는 없다.
    

- 객체지만 속성에 접근할 수 없어야 한다면, unknown 타입이 필요할 수 있다.

- 함수의 타입에도 any를 사용해서는 안 된다.

- any를 사용할 때는 정말로 모든 값이 허용되어야만 하는지 면밀히 검토해야 한다.

  

  

  

### Item 40. 함수 안으로 타입 단언문 감추기

- 함수 내부 로직이 복잡해 안전한 타입으로 구현하기 어려운 경우가 있는데, 불필요한 예외 상황까지 고여하며 타입 정보를 힘들게 구성할 필요는 없다.

- 함수 내부에는 타입 단언을 사용하고, 함수 외부로 드러나는 타입 정의를 정확히 명시하는 정도로 끝내는게 낫다.

- 프로젝트 전반에 위험한 타입 단언문이 드러나 있는 것 보다는 제대로 타입이 정의된 함수 안으로 타입 단언문을 감추는게 더 좋은 설계이다.

- 타입 단언문은 일반적으로 타입을 위험하게 만들지만, 상황에 따라 필요하거나 해결책이 될 수 있으므로 불가피하게 사용해야 한다면, 정확한 정의를 가지는 함수 내부로 숨기도록 한다.

  

  

  

### Item 41. any의 진화를 이해하기

- 타입스크립트에서 일반적으로 변수의 타입은 변수를 선언할 때 결정된다. 그 후에 정제(null 체크 해서) 될 수 있지만, 새로운 값이 추가되도록 확장하는 경우는 없다.

- 그러나 any 타입 관련 예외가 존재한다.

- any[] 로 초기화 된 값에 number 타입의 값을 넣으면 해당 값의 타입은 number[] 로 진화하게 된다.

- 배열에 다양한 타입의 요소를 넣으면 배열의 타입이 확장되며 진화한다.

- 또한, 조건문에서는 분기에 따라 타입이 변할 수 있다.

- 변수의 초기 값이 null인 경우도 any의 진화가 일어난다.

- any 타입의 진화는 noImplicitAny가 설정된 상태에서 변수의 타입이 암시적 any인 경우에만 일어난다. 그러나 명시적으로 any를 선언하면 any 타입이 그대로 유지된다.

- any 타입의 진화는 암시적 any 타입에 어떤 값을 할당할 때만 발생한다. 그리고 어떤 변수가 암시적 any 상태일 때 값을 읽으려고 하면 오류가 발생한다.

- any가 진화하는 방식은 변수가 추론되는 원리와 동일하다.

- 타입을 안전하게 지키기 위해서는 암시적 any를 진화시키는 방식보다 명시적 타입 구문을 사용하는것이 좋다.

  

  

  

  

### Item 42. 모르는 타입의 값에는 any 대신 unknown 사용하기

- unknown에는 함수의 반환값과 관련된 형태, 변수 선언과 관련된 형태, 단언문과 관련된 형태가 있다.

- 함수의 반환값과 관련된 unknown
    
    - 함수의 반환 타입으로 any는 적절하지 않은 설계인데, 함수의 반환값에 타입 선언을 강제할 수 없기 때문에 호출한 곳에서 선언을 생략하게 되면 반환 값은 암시적 any 타입이 되고, 사용하는 곳 마다 오류가 발생한다.
    
    - 대신, 반환 타입이 unknown이 되는 것이 더 안전하다.
    
    - **any의 특징으로는,**
        
        - 어떠한 타입이든 any 타입에 할당 가능하다.
        
        - never를 제외한 어떠한 타입으로도 할당 가능하다.
        
    
    - 두 특징이 있는데, 이들은 any 사용이 강력하지만 위험한 이유가 된다.
    
    - 타입 체커는 집합 기반이며, 집합 관점에서 ‘한 집합은 다른 모든 집합의 부분 집합이면서 동시에 상위 집합이 될 수 없다’는 개념에 any는 상충되기 때문에 타입 시스템에 문제를 일으키는 원인이 될 수 있다.
    
    - unknown 은 any 대신 쓸 수 있는 타입 시스템이 부합하는 타입이다.
    
    - **unknown의 특징으로는,**
        
        - 어떠한 타입이든 unknown에 할당 가능하다.
        
        - unknown은 오직 unknown과 any에만 할당 가능하다.
        
    
    - **unknown과 정 반대 타입인 never의 특징으로는,**
        
        - 어떤 타입도 never에 할당할 수 없다.
        
        - 어떠한 타입으로도 할당 가능하다.
        
    
    - unknown 상태로 값을 사용하려 하면 오류가 발생하기 때문에 적절한 타입으로 변환하도록 강제(단언)할 수 있다.
    

- 변수 선언과 관련된 unknown
    
    - 어떠한 값이 있지만, 그 타입을 모를 때 unknown 사용
    
    - 예를 들어, JSON 직렬화 가능한 타입을 예상할 수 없는 모든 것을 담는 역할을 하는 변수에 unknown 사용
    
    - instanceof 체크한 후 unknown에서 원하는 타입으로 변환 가능하다.
    
    ```tsx
    const processVal = (val: ubnknown) => {
    	if(val instanceof Date){
    		val  // 타입이 Date
    	}
    }
    ```
    
    - 사용자 정의 타입 가드도 unknown에서 원하는 타입으로 변환할 수 있다.
    
    ```tsx
    const isBook = (val: unknown): val is Book => {
    	return (
    		typeof(val)	=== 'object' && val !== null && 'name' in val && 'author' in val
    	)
    }
    
    const processVal = (val: ubnknown) => {
    	if(isBook(val){
    		val  // 타입이 Date
    	}
    }
    ```
    
    - unknown 타입의 범위를 좁히기 위해서는 상당히 많은 노력이 필요하다.
    
    - 가끔 unknown 대신 제너릭 매개변수가 사용되는 경우도 있다.
        
        - 제너릭을 사용한 스타일은 타입 단언문과 달라 보이지만 기능적으로는 동일하다.
        
        - 제너릭보다는 unknown을 반환하고 사용자가 직접 단언문을 사용하거나 원하는 대로 타입을 좁히도록 강제하는 것이 좋다.
        
    

- 단언문과 관련되 unknown
    
    - 이중 단언문에서 any 대신 unknown을 사용할 수 있다.
    
    - any의 경우, 코드를 분리하는 경우 그 영향력을 바로 알 수 없다
    
    - unknown의 경우, 코드를 분리하는 즉시 오류를 발생하게 된다.
    

- unknown과 유사하지만 조금 다른 타입
    
    - object : 비 기본형 타입으로 이루어진다. true, 12, “ff” 등이 포함되진 않지만, 객체와 배열은 포함된다.
    
    - {} : null 과 undefined 를 제외한 모든 값을 포함한다.
    
    - 최근에는 {}을 사용하는 경우가 드물며, 정말로 null과 undefined가 불가능하다고 판단되는 경우만 unknown 대신 {}을 사용한다.
    

  

  

  

### Item 43. 몽키 패치보다는 안전한 타입을 사용하기

- 객체에 임의 속성을 추가하는 것은 일반적으로 좋은 설계가 아니다.

- 예를 들어, window 또는 DOM 노드에 데이터를 추가한다고 가정했을 때, 해당 데이터는 전역 변수가 된다. 이 전역 변수는 서로 멀리 떨어진 부분들 사이에 의존성을 만들게 된다.

- 이러한 변수로 인해 함수를 호출할 때 마다 사이드 이펙트를 고려해야 한다.

- 또한, 타입스크립트는 Document와 HTMLElement 의 내장 속성에 대해서는 알고 있지만, 임의로 추가한 속성에 대해서는 알지 못한다.

- any를 사용하는 방법이 있지만, 타입 안전성 문제와 언어 서비스를 사용할 수 없다는 단점이 있다.

- 최선의 해결책은 document 또는 DOM으로부터 데이터를 분리하는 것이다.

- 분리할 수 없는 경우(객체와 데이터가 붙어 있어야만 하는 라이브러리 사용, 자바스크립트 애플리케이션 마이그레이션 과정 등), 두 가지 차선책이 있다.

- interface의 특수 기능 중 하나인 보강(agumentation)을 사용
    
    ```tsx
    export {};
    declare global {
    	interface Document {
    		monkey: string;
    	}
    }
    document.monkey = 'Tamarin';
    ```
    
    - 모듈의 관점에서(타입스크립트 파일이 import / export 를 사용하는 경우), 제대로 동작하게 하려면 global 선언을 추가해야 한다.
    
    - 보강을 사용한 방법이 any 보다 나은 점은,
        
        - 타입 안전성이 좋다
        
        - 속성에 주석을 붙일 수 있다.
        
        - 자동 완성을 사용할 수 있다.
        
        - 몽키 패치에 대한 기록이 남는다.
        
    
    - 보강을 사용할 때 주의해야 할 점은 모듈 영역(scope)과 관련이 있다.
        
        - 보강은 전역적으로 적용되기 때문에 코드의 다른 부분이나 라이브러리로부터 분리할 수 없다.
        
        - 또한, 애플리케이션이 실행되는 동안 속성을 할당하면 실행 시점에서 보강을 적용할 방법이 없다.
        
        - 특히 웹 페이지 내의 HTML 엘리먼트를 조작할 때, 어떤 엘리먼트는 속성이 있고 어떤 엘리먼트는 속성이 없는 경우 문제가 된다.
        
        - 이러한 이유로 속성을 string | undefined 로 선언할 수 도 있다.
        
    

- 더 구체적인 타입 단언문 사용
    
    ```tsx
    interface MonkeyDocument extends Document {
    	monkey: string;
    }
    (document as MonkeyDocument).monkey = 'Macaque';
    ```
    
    - MonkeyDocument 는 Document 를 확장하기 때문에 타입 단언문은 정상이며 할당문의 타입은 안전하다.
    
    - 또한 Document 를 건드리지 않고 별도로 확장하는 새로운 타입을 도입했기 때문에 모듈 영역 문제도 해결할 수 있다.(import 하는 곳의 영역에만 해당)
    
    - 따라서 몽키 패치 된 속성을 참조하는 경우에만 단언문을 사용하거나, 새로운 변수를 도입하면 된다
    

  

  

  

### Item 44. 타입 커버리지를 추적해 타입 안전성 유지하기

- noImplicitAny 설정을 추가하고 모든 암시적 any 대신 명시적 타입 구문을 추가해도 any 타입과 관련된 문제들로부터 안전하다고 할 수 없다.

- any 타입이 프로그램 내에 여전히 존재할 수 있는 두 가지 경우가 있다.
    
    - 명시적 any
    
    - 서드파티 타입 선언
    

- 패키지를 활용해 any 타입을 추적할 수 있다.
