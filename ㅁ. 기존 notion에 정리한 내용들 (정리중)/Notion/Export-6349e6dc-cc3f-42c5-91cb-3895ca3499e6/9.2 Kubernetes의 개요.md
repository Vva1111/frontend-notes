분산 시스템의 자동화된 배포, 확장, 관리 등을 위한 오픈소스 플랫폼으로, 복잡한 애플리케이션의 관리와 운영을 단순화하고, 효율적인 자원 관리를 가능하게 한다.

> 여러 서버들에서 컨테이너 관리  
> 컨테이너 간 네트워크 관리  
> 컨테이너의 부하분산  
> 컨테이너의 감시  
> 무정지로 업데이트

- Kubernetes 주요 기능 (From. ChatGPT)
    
    ### 1. **자동화된 컨테이너 배포 (Automated Container Deployment)**
    
    - Kubernetes는 컨테이너를 자동으로 배포하고, 롤링 업데이트 및 롤백을 통해 애플리케이션의 새로운 버전을 무중단으로 배포할 수 있습니다.
    
    - 또한, **배포 전략**(롤링 업데이트, 블루-그린 배포 등)을 선택하여 애플리케이션을 효율적으로 업데이트할 수 있습니다.
    
    ### 2. **자동 스케일링 (Automatic Scaling)**
    
    - **Horizontal Pod Autoscaling (HPA)**: Kubernetes는 클러스터 내에서 리소스 사용량(CPU, 메모리 등)을 기반으로 자동으로 Pod(컨테이너 단위)를 확장하거나 축소할 수 있습니다.
    
    - **Cluster Autoscaler**: 전체 클러스터의 노드 수를 동적으로 조정하여 자원을 효율적으로 관리합니다.
    
    - **Vertical Pod Autoscaling (VPA)**: Pod의 리소스를 동적으로 조정하여, 애플리케이션이 필요로 하는 리소스를 적절히 할당합니다.
    
    ### 3. **고가용성 (High Availability)**
    
    - Kubernetes는 **복제와 셀프 힐링**을 통해 고가용성을 제공합니다. 만약 어떤 Pod가 실패하면, 자동으로 새로운 Pod가 재시작되거나 다시 배포됩니다.
    
    - **ReplicaSet**을 사용해 여러 개의 복제본을 유지하여, 서비스의 안정성과 가용성을 보장합니다.
    
    ### 4. **서비스 디스커버리와 로드 밸런싱 (Service Discovery & Load Balancing)**
    
    - Kubernetes는 자동으로 각 서비스에 대한 DNS 이름을 할당하고, 클러스터 내에서 트래픽을 자동으로 분산시켜 **로드 밸런싱**을 제공합니다.
    
    - 클러스터 외부에서 접근할 수 있는 **Ingress** 리소스를 설정하여 외부 트래픽을 적절한 서비스로 라우팅합니다.
    
    ### 5. **스토리지 관리 (Storage Management)**
    
    - Kubernetes는 다양한 스토리지 시스템(Amazon EBS, Google Cloud Persistent Disk 등)을 지원하여, 컨테이너가 실행되는 노드에 데이터가 영구적으로 저장될 수 있도록 합니다.
    
    - *Persistent Volumes (PV)**와 **Persistent Volume Claims (PVC)**를 사용하여 스토리지 자원을 관리하고, 동적으로 스토리지를 프로비저닝할 수 있습니다.
    
    ### 6. **네트워킹 (Networking)**
    
    - Kubernetes는 Pod 간의 **통신을 관리**하고, 네트워크 정책을 설정하여 보안을 강화합니다.
    
    - *CNI(Container Network Interface)**를 통해 다양한 네트워크 플러그인(예: Calico, Flannel)을 지원합니다.
    
    - **Service**를 통해 클러스터 내에서 동적인 IP 주소를 할당하고, 네트워크를 통해 서비스 간의 통신을 관리합니다.
    
    ### 7. **비밀 관리 및 구성 관리 (Secrets & ConfigMaps)**
    
    - Kubernetes는 애플리케이션의 **비밀 정보**(예: API 키, 비밀번호)나 **구성 파일**을 **Secret**과 **ConfigMap** 리소스를 통해 안전하게 관리합니다.
    
    - 이를 통해 민감한 정보를 코드와 분리하여 보안성을 높이고, 환경 변수로 쉽게 활용할 수 있습니다.
    
    ### 8. **롤링 업데이트와 롤백 (Rolling Updates & Rollbacks)**
    
    - Kubernetes는 애플리케이션의 새로운 버전을 **무중단 배포**(롤링 업데이트)하여, 시스템의 가용성을 유지합니다.
    
    - 배포 중 문제가 발생하면, **자동 롤백** 기능을 통해 이전 버전으로 되돌릴 수 있습니다.
    
    ### 9. **모니터링과 로깅 (Monitoring & Logging)**
    
    - Kubernetes는 애플리케이션의 상태를 실시간으로 모니터링하고, **Prometheus**와 **Grafana**와 같은 도구를 통해 시스템 메트릭을 수집합니다.
    
    - **Fluentd**, **Elasticsearch**, **Kibana**와 같은 툴을 통해 로그를 집계하고 분석할 수 있습니다.
    
    ### 10. **자원 요청 및 제한 (Resource Requests & Limits)**
    
    - Kubernetes는 각 컨테이너에 대해 **CPU와 메모리 자원의 요청**과 **제한**을 설정할 수 있습니다. 이를 통해 시스템의 자원을 효율적으로 관리하고, 과다한 자원 사용을 방지할 수 있습니다.
    
    ### 11. **네임스페이스 (Namespaces)**
    
    - Kubernetes는 **네임스페이스**를 사용하여 클러스터 내의 자원들을 논리적으로 분리하고, 서로 다른 환경(예: 개발, 테스트, 운영)을 구분할 수 있습니다.
    
    - 네임스페이스를 사용하면 **리소스 할당 및 권한 관리**를 세밀하게 할 수 있습니다.
    
    ### 12. **확장성과 유연성 (Scalability & Flexibility)**
    
    - Kubernetes는 수천 개의 노드와 수만 개의 컨테이너를 관리할 수 있는 뛰어난 **확장성**을 제공합니다.
    
    - 다양한 클라우드 제공자(예: AWS, GCP, Azure)와 온프레미스 환경에서 유연하게 사용할 수 있습니다.

## Kubernetes의 서버 구성

![[IMG_5525272B99CB-1.jpeg]]

> [!important]
> 
> ### **마스터 서버 ( = 컨트롤플레인)**
> 
> Kubernetes 클러스터 안의 컨테이너를 조작하기 위한 서버
> 
> 여러대로 구성된 kubernetes 클러스터 안에 있는 노드의 리소스 사용 상황을 확인하고 , 컨테이너를 시작할 노드를 자동으로 선택한다. 마스터 서버가 여러 대로 분산 구성된 노드를 모아서 관리함으로써 이용자가 봤을 때는 마치 한 대의 서버인 것처럼 행동한다.

> [!important]
> 
> ### 백엔드 데이터베이스(etcd)
> 
> etcd라 부르는 분산 키 밸류 스토어(KVS)를 사용하여 클러스터의 구성 정보를 관리한다.
> 
> 클러스터를 구축하기 위한 설정 정보가 들어있다.
> 
> 시스템 구성에 따라 마스터 서버 상에 구축하는 경우도 있다.

> [!important]
> 
> ### 워크 노드(Node)
> 
> 워크 노드(Worker Node)는 Kubernetes 클러스터에서 실제 애플리케이션을 실행하는 서버
> 
> 워크 노드는 파드를 실행하고, 그 안에 포함된 컨테이너를 관리
> 
> 컨트롤 플레인은 워크 노드를 관리하고, 파드를 어디에 배치할지 결정
> 
> 클라우드 환경에서는 가상 머신이 워크 노드 역할

## 애플리케이션 구성 관리(Pod, ReplicaSet, Deployment)

> [!important]
> 
> ### 파드 Pod
> 
> ![[IMG_A106865EF598-1.jpeg]]
> 
> 여러 개의 컨테이너를 모아서 Pod로 관리
> 
> 노드에는 여러 개의 Pod가 배치되며, Pod는 반드시 동일한 노드 상에 동시에 전개된다.
> 
> Pod는 애플리케이션의 전개 단위로, Pod 단위로 컨테이너를 작성, 시작, 정지, 삭제하는 조작을 수행한다.
> 
> Pod 안의 여러 컨테이너는 가상 NIC(프라이빗 IP)를 공유하여, 컨테이너끼리 Localhost 경유로 통신할 수 있다.

> [!important]
> 
> ### 리플리카 셋 ReplicaSet
> 
> ![[IMG_D1A1868F2CB3-1.jpeg]]
> 
> K8s 클러스터에서 미리 지정된 수의 파드를 항상 실행시키는 장치  
> ⇒ 즉, ReplicaSet은 클러스터 내에서 정해진 수의 파드가 항상 실행되도록 보장
> 
> 리플리카 수는 클러스터에 실행할 파드의 개수를 의미하며, ReplicaSet을 통해 설정된다.
> 
> ReplicaSet은 이 수를 관리하여, 설정된 수의 파드가 항상 실행되도록 유지한다. 이를 통해 애플리케이션의 가용성을 보장하며, 파드가 실패할 경우 새로운 파드를 자동으로 생성해 리플리카 수를 유지한다.
> 
> Pod의 수를 동적으로 변경하여 오토스케일을 구현할 수도 있다.

> [!important]
> 
> ### 디플로이먼트, 배포 Deployment
> 
> ![[IMG_92B3D67905EE-1.jpeg]]
> 
> 디플로이먼트(Deployment)는 애플리케이션의 파드를 자동으로 생성, 업데이트, 롤백할 수 있다.
> 
> - **파드의 생성 및 관리**: 설정된 **리플리카 수** 만큼 파드를 실행시킨다.
> 
> - **자동 업데이트**: 새로운 버전의 애플리케이션을 배포할 때 **무중단**으로 업데이트를 수행한다.
> 
> - **롤백**: 배포한 애플리케이션에 문제가 있을 경우, 이전 버전으로 **롤백**이 가능하다.

> [!important]
> 
> ### **그 외…**
> 
> 데몬셋(DaemonSet): 모든 노드에 하나의 파드를 배포하여 지속적으로 실행.
> 
> 잡(Job): 일회성 작업을 실행하고 완료된 후 종료.
> 
> 크론잡(CronJob): 주기적인 작업을 크론 표현식에 따라 자동 실행.

## 네트워크 관리(Service)

![[IMG_312F48B0CC14-1.jpeg]]

Pod에 대해 외부로부터 액세스할 때는 서비스를 정의한다.

- Load Balancer : Service에 대응하는 IP주소 + 포트 번호  
    → 여러 Pod에 대한 레이어 4 레벨의 부하분산이 일어남

- ClusterIP : 클러스터 내부에서만 접근 가능한 고정 IP 주소  
    → 클러스터 내의 다른 파드들이 해당 서비스를 통해 네트워크 통신을 할 수 있도록 한다. 외부에서 이 IP에 접근할 수는 없으며, 서비스는 클러스터 내에서만 사용

- ExternalIP : Kubernetes 서비스가 외부에서 접근할 수 있도록 해주는 IP 주소  
    → ExternalIP는 클라우드 제공자의 로드밸런서 또는 물리적인 네트워크 환경에서 제공되는 외부 IP 주소일 수 있다. 이 주소는 외부 네트워크에서 클러스터로 접근할 수 있게 함

## Label을 사용한 리소스 식별

**Label**은 Kubernetes 리소스를 식별하고 **그룹화**하는 데 사용하는 key-value 쌍이다.

이를 통해 **파드, 서비스, 디플로이먼트** 등의 리소스를 쉽게 선택하고 필터링할 수 있다.

Label이 붙은 리소스를 참조하려면 selector로 지정한다.

![[IMG_C4E1CEE401E8-1.jpeg]]

==* 예시==

_==Pod에 label 붙이는 코드==_

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
  labels:
    app: my-app       # 라벨 key: app, value: my-app
    environment: prod  # 라벨 key: environment, value: prod
spec:
  containers:
  - name: nginx
    image: nginx
```

  

_==Deployment나 Service를 정의할 때, 특정 레이블을 가진 파드만 선택하려면 selector를 사용하여 이를 지정==_

```yaml
selector:
  matchLabels:
    app: my-app
```

## Kubernetes의 구조

![[IMG_2045F2A3D581-1.jpeg]]

Kubernetes 컴포넌트 개요

### ==1. Control Plane (구 Master)==

==**API Server**==

Kubernetes의 리소스 정보를 관리하기 위한 프론트엔드 REST API

각 컴포넌트로부터 리소스 정보를 받아 etcd에 저장한다. 다른 컴포넌트는 API Server를 통해 etcd 정보에 액세스한다.

==**Scheduler**==

Pod를 어떤 노드에서 작동시킬지 제어하는 백엔드 컴포넌트

노드에 할당되어 있지 않은 Pod에 대해 Kubernetes 클러스터 상태 확인 후 빈 영역을 가진 노드를 찾아 Pod를 실행시키는 스케줄링을 수행한다.

==**Controller Manager**==

클러스터의 상태를 항상 감시하는 백엔드 컴포넌트

정의파일에서 정의한 것과 실제 노드나 컨테이너에서 움직이고 있는 상태를 모아서 관리한다.

### 2. etcd

클러스터 구성을 유지 관리하는 KVS(Key-Value Store)이다. 어떤 Pod를 어떻게 배치할지와 같은 구성정보를 갖고있으며, API Server가 참조한다.

### 3. Node

==**kubelet**==

노드에서는 kubelet라는 에이전트가 작동하고 있다.

Pod의 정의 파일에 따라 Docker 컨테이너를 실행하거나 스토리지를 마운트 하는 기능을 갖고있다.

각 노드의 상태를 지속적으로 모니터링하고, **노드의 상태**가 바뀌면, 이를 API 서버에 통지한다.

manifest파일로 클러스터 구성 정보를 관리할 수 있다.

![[IMG_54DBB8AEC5B0-1.jpeg]]
