### 타입 스크립트란 무엇인지, 자바스크립트와는 어떤 관계인지, null, any, duck typing 등에 대해 알아본다.

  

- 타입 스크립트는 사용 방식 면에서 조금은 특이한 언어이다.

- 직접 인터프리터 실행 혹은 컴파일 되지 않으며, 또 다른 고 수준 언어인 자바스크립트로 컴파일 되고, 실행 또한 자바스크립트로 이루어진다.

  

### Item 1. 타입 스크립트와 자바스크립트의 관계 이해

- 타입 스크립트는 자바스크립트의 슈퍼셋(상위 집합) 언어이다. ( 문법적으로도 )

- 자바스크립트와 타입 스크립트는 서로 다른 확장자를 사용하지만, 이것이 두 언어가 완전히 다른 언어임을 뜻하는 것은 아니다.

- 타입 스크립트는 자바스크립트의 상위 집합이기 때문에 .js 파일에 있는 코드는 이미 타입 스크립트라고 할 수 있다.

- 이러한 특성은 기존에 존재하던 자바스크립트 코드를 타입 스크립트로 마이그레이션 하는데 엄청난 이점이 될 수 있다.

- 모든 자바스크립트 프로그램이 타입 스크립트라는 명제는 참이지만, 그 반대는 성립하지 않는다.타입스크립트이지만 자바스크립트가 아닌 프로그램이 존재할 수 있기 때문이다.
    
    - 이는 타입 스크립트가 타입을 명시하는 자바스크립트에 존재하지 않는 추가적인 문법을 가지고 있기 때문이다.
    

```jsx
function greet(who: string){
	console.log('hello', who)
}
```

- 위의 코드를 예로 들자면, 타입 스크립트에서는 문제 없는 코드이지만, 순수 자바스크립트에서는 (who: sting) 부분에서 오류를 출력하게 된다.
    
    - :string 과 같은 문법은 타입 스크립트에서 쓰이는 타입 구문이다.
    
    - 타입 구문을 사용하는 순간부터 자바스크립트는 타입 스크립트의 영역으로 들어간다.

    - 타입 스크립트가 ‘정적’ 타입 시스템이라고 불리는 것은 이러한 특징을 말하는 것이다.

- 타입 스크립트는 타입 구문 없이도 오류를 찾아낼 수 있지만, 타입 구문을 추가한다면 더 많은 오류를 찾아낼 수 있다. 코드의 ‘의도’가 무엇인지 타입 구문을 통해 타입 스크립트에게 알려줄 수 있기 때문이다.

  

- 타입 스크립트 타입 시스템은 자바스크립트의 런타임 동작을 ‘모델링’한다.
    
    - 이 말은, 단순히 런타임 동작을 모델링 하는 것 뿐만 아니라, 의도치 않은 이상한 코드가 오류로 이어질 수도 있다는 뜻이다.(모든 오류를 찾아낼 수는 없고, 타입 체커를 통과해도 런타임에서 오류를 발생 시킬 코드는 충분히 존재할 수 있다.)
    
    - 하지만, 타입 스크립트의 도움을 받으면 오류가 적은 코드를 작성할 수 있고,
    
    - null+7, []+12 혹은 불필요한 매개변수를 추가해서 함수 호출하는 것을 당연하게 여긴다면 차라리 타입 스크립트를 사용하지 않는 것이 나을 수 있다.
    

  

- 타입 시스템이 정적 타입의 정확성을 보장해줄 것 같지만 그렇지 않다.
    
    - 애초에 타입 시스템은 그런 목적으로 만들어지지 않았다.
    

  

### Item 2. 타입 스크립트 설정 이해

- 현재 시점으로 타입 스크립트 컴파일러는 약 100여 개의 설정 값을 가지고 있다.

- 설정은 커맨드 라인 혹은 설정 파일로 사용할 수 있지만, 가급적 설정 파일 사용이 권장된다.
    
    - 설정 파일 생성은 **ts —init** 실행
    

- 설정을 제대로 사용하기 위해서는 **noImplicitAny**와 **strictNullCheck** 를 이해해야 한다.
    
    1. **noImplicitAny :** 변수들이 미리 정의된 타입을 가져야 하는지 여부를 제어
    
    ```jsx
    function add(a, b){
    	return a + b;
    }
    
    function add(a: number, b: number){
    	return a + b;
    }
    ```
    
    - 위 코드는 매개변수에 any를 명시하지 않았지만 any 타입으로 간주되기 때문에 이를 암시된(암묵적으로 합의된) any 라고 한다.
    
    - 타입 스크립트는 정보를 가질 때 가장 효과적이기 때문에, 되도록이면 **noImplicitAny** 를 설정하고 코드를 작성할 때 마다 타입을 명시하도록 해야 한다.
    
    - 이는 타입 스크립트가 문제를 더 쉽게 발견할 수 있게 하고, 코드의 가독성을 높일 수 있기 때문에 개발자의 생산성이 향상될 수 있다.
    
      
    
    1. **strictNullCheck :** null과 undefined가 모든 타입에서 허용되는지 확인하는 설정
    
    ```jsx
    const x: number = null;  // 정상
    const x: number = null;  // 오류 - 'null' 형식은 'number' 형식에 할당할 수 없습니다.
    ```
    
    - 위 코드가 **strictNullCheck** 를 해제 했을 때, 아래가 설정했을 때 이다.
    
    - 이 설정은 null 혹은 undefined 관련 오류를 잡아내는 데 많은 도움이 되지만, 코드 작성을 어렵게 한다.
    

  

### Item 3. 코드 생성과 타입이 관계 없음을 이해하기

- 큰 그림에서 보면, 타입 스크립트 컴파일러는 두 가지 역할을 수행한다.
    
    1. 최신 타입 스크립트/자바스크립트를 브라우저에서 동작할 수 있도록 구 버전의 자바스크립트로 트랜스파일 한다.
    
    1. 코드 타입 오류를 체크한다.
    
    - 위 두 가지 역할은 완벽하게 독립적으로 이루어진다.
    
    - 타입 스크립트가 자바스크립트로 변환될 때 코드 내의 타입에는 영향을 주지 않는다.
    
    - 또한 그 자바스크립트의 실행 시점에도 타입은 영향을 미치지 않는다.
    

  

- 타입 오류가 있는 코드도 컴파일이 가능하다.
    
    - 컴파일은 타입 체크와는 독립적으로 이루어지기 때문에 타입 오류가 있는 코드도 컴파일이 가능하다.
    
    - 타입 스크립트 오류는 C 혹은 자바 같은 언어들의 경고(Warning)와 비슷하다. 문제를 알려주지만, 그렇다고 빌드를 멈추지는 않는다는 것이다.
    
    - 만약 오류가 있을 때, 컴파일 하지 않으려면 설정 파일의 **noEmitOnError** 를 설정하거나 빌드 도구에 동일하게 적용하면 된다.
    

  

- 런타임에는 타입 체크가 불가능하다
    
    - 타입 스크립트의 타입은 ‘제거 가능’하다
    
    - 실제로 자바스크립트로 컴파일 되는 과정에서 모든 인터페이스, 타입, 타입 구문은 그냥 제거되어 버린다.
    
    - 런타임에 타입 정보를 유지하는 방법은 해당 속성이 있는지 체크하거나, 런타임에 접근 가능한 타입 정보를 명시적으로 저장하는 ‘태그’ 기법, 타입을 클래스로 만드는 방법 등이 있다.
    

  

- 타입 연산은 런타임에 영향을 주지 않는다.
    
    ```jsx
    function asNumber(val: number | string): number{
    	return val as number;
    }
    ```
    
    - string 또는 number 타입인 값을 항상 number로 정제하는 위와 같은 코드의 경우,타입 체커는 통과할 수 있지만, 잘못된 방법을 사용한 것이다.
    
    - 위 코드는 자바스크립트로 변환되면서 타입 정보를 모두 잃기 때문에 코드에 아무런 정제 과정이 없어지게 된다. as number는 타입 연산이고, 이는 런타임 동작에는 아무런 영향을 미치지 않기 때문이다.
    
    - 값을 정제하기 위해서는 런타임의 타입을 체크해야 하고, 자바스크립트 연산을 통해 변환을 수행해야 한다.
    
    ```jsx
    function asNumber(val: number | string): number{
    	return typeof(val) === 'string' ? Number(val) : val;
    }
    ```
    

  

  

- 런타임 타입은 선언된 타입과 다를 수 있다.
    
    - 타입 스크립트에서는 런타임 타입과 선언된 타입이 맞지 않을 수 있다. 타입이 달라지는 혼란스러운 상황을 가능한 한 피해야 한다. 선언된 타입이 언제든지 달라질 수 있다는 것을 명심해야 한다.
    

  

- 타입 스크립트 타입으로는 함수를 오버로드 할 수 없다.
    
    - 타입 스크립트가 함수 오버로딩 기능을 지원하기는 하지만, 온전히 타입 수준에서만 동작한다
    
    - 하나의 함수에 대해 여러 개의 선언문을 작성할 수 있지만, 구현체는 오직 하나이다.
    

  

- 타입 스크립트 타입은 런타임 성능에 영향을 주지 않는다.
    
    - 타입과 타입 연산자는 자바스크립트 변환 시점에 제거되기 때문에 런타임의 성능에 아무런영향을 주지 않는다.
    
    - 타입 스크립트의 정적 타입을 실제로 비용이 전혀 들지 않는다.
    
    - 런타임 오버헤드가 없는 대신, 빌드타임 오버헤드가 있다. 빌드 오버헤드가 커지면, 빌드 도구에서 ‘transfile only’ 설정을 통해 타입 체크를 건너뛸 수도 있다.
    

  

  

### Item 4. 구조적 타이핑에 익숙해지기

- 자바스크립트는 본질적으로 ‘duck typing’ 기반이다. 이는 만약 어떤 함수의 매개변수 값이 모두 제대로 주어진다면, 그 값이 어떻게 만들어졌는지 신경 쓰지 않고 사용한다는 의미이다.

- (옮긴이) 객체가 어떤 타입에 부합하는 변수와 메서드를 가질 경우, 객체를 해당 타입에 속하는 것으로 간주하는 방식, “만약 어떤 새가 오리처럼 걷고, 헤엄치고, 꽥꽥거릴 경우, 그 새를 오리라고 부를 것이다”라는 명제로 정의하는 The Duck Test 에서 유래

- 타입 스크립트는 이러한 동작 방식을 그대로 모델링한다.

- 어떤 인터페이스에 할당 가능한 값이라면 타입 선언에 명시적으로 나열된 속성들을 가지고 있을 것이다 ( 타입은 ‘봉인’되어 있지 않다. )

  

### Item 5. any 타입 지양하기

```jsx
let age: number;
age = '12';  // '12'형식은 number 형식에 할당할 수 없다.
age = '12' as any;  // OK
```

- 일부 특별한 경우를 제외하고는 any를 사용할 경우, 타입 스크립트의 이점을 누릴 수 없게 된다.

- any type에는 타입 안전성이 없다.
    
    - 위 코드에서 age는 number 타입으로 선언되었지만, as any를 사용하는 경우 age에 string을 할당할 수 있게 된다. 타입 체커는 선언에 따라 number 타입으로 판단할 것이기 때문에 타입에 혼란이 오게 된다.
    

  

- any는 함수 시그니처를 무시한다.
    
    - 함수를 호출하는 쪽은 약속된 타입의 입력을 제공하고, 함수는 약속된 타입의 출력을 반환한다.
    
    - 그러나 any의 사용은 이런 약속을 어길 수 있다.
    

  

- any 타입에는 언어 서비스가 적용되지 않는다.
    
    - 어떤 심벌에 타입이 있다면, 타입 스크립트 언어 서비스는 자동 완성 기능, 이름 변경 기능(vscode - f2)과 적절한 도움말 등을 제공할 수 있다.
    
    - 그러나 any 타입인 심벌을 사용하면 아무런 도움을 받을 수 없게 된다.
    

  

- any 타입은 코드 리팩토링 때 버그를 감춘다.

- any는 타입 설계를 감춘다.

- any는 타입 시스템의 신뢰도를 떨어트린다.
    
    - any를 사용할 경우, 런타임에 오류를 발견할 확률이 높기 때문에 신뢰성이 떨어진다. 정확한 타입을 설정해 런타임 이전에 미리 오류를 잡아내 신뢰도를 높여야 한다.
