### Item 53. 타입스크립트 기능보다는 ECMAScript 기능 사용하기

- TC39(타입스크립트 표준 기구)는 런타임 기능을 발전시키고, 타입스크립트 팀은 타입 기능만 발전시킨다는 명확한 원칙이 있다.

- 하지만 이 원칙이 세워지기 전에 이미 사용되고 있던 기능들이 있다. 아래 기능들은 타입 공간과 값 공간(자바스크립트)의 경계를 혼란스럽게 만들기 때문에 사용하지 않는 것이 좋다.
    
    - 열거형(enum)
        
        - 단순히 값을 나열하는 것보다는 실수가 적고 명확하기 때문에 열거형을 사용하는 것이 좋지만, 타입스크립트의 열거형은 몇 가지 문제가 있다.
        
        - 타입스크립트의 열거형은 아래 목록처럼 상황에 따라 다르게 동작한다.
            
            - 숫자 열거형에 명시된 값 외의 다른 숫자가 할당되면 매우 위험하다.
            
            - 상수 열거형은 보통의 열거형과 달리 런타임에 완전히 제거된다.
            
            - preserveConstEnum 플래그를 설정한 상태의 상수 열거형은 보통의 열거형처럼 런타임 코드에 상수 열거형 정보를 유지한다.
            
            - 문자열 열거형은 런타임의 타입 안전성과 투명성을 제공한다. 그러나 타입스크립트의 다른 타입과 달리 구조적 타이핑이 아닌 명목적 타이핑을 사용한다.  
                구조적 타이핑은 구조가 같으면 할당이 허용되는 반면, 명목적 타이핑은 타입의 이름이 같아야 할당이 허용된다.
            
        
        - 자바크스립트와 타입스크립트에서의 동작이 다르기 때문에 사용하지 않는 것이 권장된다.
        
        - 열거형 대신 리터럴 타입의 유니온을 사용하는 것이 좋다.
        
        ```tsx
        type Faaa = 'aa' | 'bb' | 'cc';
        
        let faaa: Faaa = 'bb';    // 정상
        ```
        
    
    - 매개변수 속성
        
        - 일반적으로 클래스를 초기화할 때 속성을 할당하기 위해 생성자의 매개변수를 사용한다.
        
        ```tsx
        class Person {
        	name: string;
        	constructor(name: string){
        		this.name = name;
        	}
        }
        ```
        
        - 타입스크립트는 이보다 더 간결한 문법을 제공한다.
        
        ```tsx
        class Person{
        	constructor(public name: string) {}
        }
        ```
        
        - public name은 ‘매개변수 속성’ 이라고 불리며, 멤버 변수로 name을 선언한 이전 예제와 동일하게 작동한다.
        
        - 하지만 매개변수 속성은 아래와 같은 문제점이 존재한다.
            
            - 일반적으로 타입스크립트 컴파일은 타입 제거가 이루어지므로 코드가 줄어들지만, 매개변수 속성은 코드가 늘어나는 문법이다.
            
            - 매개변수 속성이 런타임에는 실제로 사용되지만. 타입스크립트 관점에서는 사용되지 않는 것처럼 보인다.
            
            - 매개변수 속성과 일반 속성을 섞어서 사용하면 클래스의 설계가 혼란스러워진다..
            
        
        - 매개변수 속성을 사용하는 것이 좋은지에 대해서는 찬반 논란이 있다.
        
        - 하지만, 매개변수 속성은 타입스크립트의 다른 패턴들과는 이질적이고, 초급자에게 생소한 문법이다.
        
    
    - 네임스페이스와 트리플 슬래시 임포트
        
        - ECMAScript 2015 이전에 타입스크립트가 자체적으로 구축한 모듈 시스템인 module 키워드와 ‘트리플 슬래시’ 임포드 방식이 있다.
        
        - ECMAScript 2015가 공식적으로 모듈 시스템을 도입한 이후, 타입스크립트는 module과 같은 기능을 하는 namespace 키워드를 추가했다.
        
        - 트리플 슬래시 임포트와 module 키워드는 호환성을 위해 남아있을 뿐이며, 이제는 ECMAScript 2015 스타일의 모듈(import, export)을 사용해야 한다.
        
    
    - 데코레이터
        
        - 데코레이터는 클래스, 메서드, 속성에 애너테이션을 붙이거나 기능을 추가하는데 사용할 수 있다.
        
        - 예를 들어, 클래스의 메서드가 호출될 때마다 로그를 남기려면 logged 애너테이션을 정의할 수 있는 식이다.
        
        - 데코레이터는 앵귤러 프레임워크를 지원하기 위해 추가되었으며, 표준화가 완료되지 않았기 때문에, 사용 중인 데코레이터가 비표준으로 바뀌거나 호환성이 깨질 가능성이 있다.
        
    

  

  

  

### Item 54. 객체를 순회하는 노하우

- 객체를 다룰 때에는 항상 ‘프로토타입 오염’의 가능성을 염두에 두어야 한다.

- Object.prototype 에 순회 가능한 속성을 절대로 추가하면 안된다.

- 객체를 순회하며 키와 값을 얻으려면, keyof 선언이나, Object.entries를 사용하면 된다.
    
    - keyof 선언은 상수이거나, 추가적인 키 없이 정확한 타입을 원하는 경우에 적절하다.
    
    - Object.entries의 경우, 더욱 일반적으로 사용되기는 하지만, 키와 값의 타입을 다루기 까다롭다.
    

- 객체를 순회할 때, 키가 어떤 타입인지 정확히 파악하고 있다면 let k: keyof T와 for-in 루프를 사용할 수 있다.

- 객체를 순회하며 키와 값을 얻는 가장 일반적인 방법은 Object.entries를 사용하는 것이다.

  

  

  

### Item 55. DOM 계층 구조 이해하기

- DOM 계층은 웹브라우저에서 자바스크립트를 실행할 때, 어디에서나 존재한다.

- 타입스크립트에서는 DOM 엘리먼트의 계층 구조를 파악하기 용이하다.

- 아래는 DOM 엘리먼트의 계층구조에 따른 타입의 몇 가지 예시이다.
    
    - EventTarget : DOM 타입 중 가장 추상화된 타입이다. 이벤트 리스너를 추가하거나 제거하고, 이벤트를 보내는 것밖에 할 수 없다.
        
        - event의 currentTarget 속성의 타입은 EventTarget | null 이다. 이 타입은 실제로 HTMLElement 타입이지만, 타입 관점에서는 window 혹은 XMLHttpRequest가 될 수도 있다는 것을 주의해야 한다.
        
    
    - Node : Element가 아닌 Node인 경우를 예로 들자면, 텍스트 조각과 주석이 있다.
    
    - Element, HTMLElement
    
    - HTMLxxxElement : 이 형태의 특정 엘리먼트들은 자신만의 고유 한 속성을 가지고 있다. 이러한 속성에 접근하려면 타입 정보 역시 실제 엘리먼트 타입이어야 하므로 상당히 구체적인 타입을 지정해야 한다.
    

- 일반적으로 타입 단언문은 지양해야 하지만, DOM 관련 타입은 타입스크립트보다 사용자가 더 잘 파악하고 있을 수 있으므로, 단언문을 사용해도 괜찮다.

  

  

  

### Item 56. 정보를 감추는 목적으로 Private 사용하지 않기

- 타입스크립트에는 public, protected, private 접근제어자를 사용할 수 있지만, 이들 또한 타입 키워드이기 때문에 컴파일 후에는 제거된다.

- 타입스크립트 코드는 컴파일 과정에서 자바스크립트 코드로 변환되며, 이 때 접근제어자는 런타임에 아무런 효력이 없다.

- 타입스크립으틔 접근제어자는 컴파일 시점에만 오류를 표현해줄 뿐이다.

- 따라서, 정보를 감추기 위해 private 키워드를 사용하면 안된다.

- 자바스크립트에서 정보를 숨기기 위해 가장 효과적인 방법은 클로져(closure)를 사용하는 것이다.
    
    - 하지만, 일반적인 객체지향 언어에서는 동일 클래스의 개별 인스턴스끼리 private 속성 접근이 가능(클래스 단위 비공개)하지만, 클로져 방식은 동일 클래스의 개별 인스턴스 간 속성 접근이 불가능(인스턴스 단위 비공개)하기 때문에 철저하게 비공개이면서 동시에 불편함이 따르게 된다.
    
    - 또한, 인스턴스를 생성할 때마다 각 메서드의 복사본이 생성되기 때문에 메모리를 낭비할 수 있다.
    

- 또 다른 선택지로는 표준화 진행 중인 비공개 필드(#) 기능을 사용할 수 있다.
    
    - 이 방법은 클로져 기법과는 다르게 클래스 메서드나 동일한 클래스의 개별 인스턴스에서는 접근이 가능하다.
    

  

  

  

### Item 57. 소스맵을 사용해 타입스크립트 디버깅하기

- 타입스크립트 코드를 실행한다는 것은, 즉 타입스크립트 컴파일러가 생성한 자바스크립트 코드를 실행한다는 말이다. 이렇게 변환된 자바스크립트 코드는 복잡해 디버깅하기 매우 어렵다.

- 디버깅 문제를 해결하기 위해 브라우저 회사들이 협력해 소스 맵(source map)이라는 해결책을 내놓았다.

- 소스맵을 사용해 런타임에 타입스크립트를 디버깅 할 수 있도록 한다.
