- 어차피 바뀔 값인데 let을 사용하는게 아니라 const 를 사용해 상수선언을 한 뒤, .value 값으로 값 할당을 변경하는 이유?

#### 사용 원인
- 자바스크립트의 언어적 특성과 Vue 엔진의 반응성 시스템이 원인

- **const 는 '반응형 객체' 자체를 보호하기 위한 안전장치**
	
	자바스크립트에서 const 는 변수 값이 아닌 참조 메모지 주소를 변경하지 못하게 막는 것이다.
	
	ref(0) 을 호출하면 Vue 는 숫자 0 을 감싼 하나의 **반응형 객체(Reactive Object)** 를 반환한다.
	
	따라서, const를 사용해 '이 변수는 영원히 Vue가 만든 이 반응형 객체만 바라본다' 라고 강제시키고(메모리 참조 주소 고정), 해당 반응성 객체 내부의 실제 값의 변경은 반응성 객체의 내부 속성인 .value를 사용해 안전하게 변경한다.


- **Vue가 변수의 변화를 감지하고 렌더링을 하려면, 값을 읽고 쓸 때 알람을 알려주는 감지기(Getter/Setter)가 필요하다.**
	
	하지만 자바스크립트의 원시 값은 이런 감지기를 직접 달아줄 수 없기 때문에 .value 라는 속성을 가진 (반응성)객체로 한 번 포장을 해주는 과정이 필요하다. 객체로 포장을 해 두면, 값의 변경이 생길 때, 객체의 Setter로 렌더링에 필요한 신호를 보낼 수 있게 되는 것이다.


- **반응성의 유실 없이 이곳저곳으로 전달하기 위해**
	자바스크립트에서 원시 값은 다른 함수나 컴포넌트로 전달될 때, **값 자체** 만 복사 되어 넘어가기 때문에 원본과의 연결 고리가 끊어진다.
	
	하지만, ref로 감싼 객체 형태로 값을 전달하면 객체의 메모리 주소(참조 값)가 전달(Call by Reference)되므로, 다른 컴포넌트나 공통 유틸 함수 내부에서 .value를 수정하더라도 원본의 반응성이 그대로 유지된다.
	
	여러 컴포넌트나 복잡한 아키텍쳐 환경에서 상태를 공유할 때 이 방식은 필수적이다.


- **결론**
	const는 Vue가 만들어 둔 반응성 객체와의 연결 고리를 영구적으로 유지하기 위해 사용하는 것이다.
	
	const로 변수의 메모리를 참조하는 메모리 주소 값은 상수(고정)화 시켜 놓고, .value를 사용해 해당 메모리의 변수 값을 변경하고, 그 변화를 반응성 객체의 Setter로 처리해 그에 따른 렌더링을 적용한다.

- 예시
	let을 사용해 반응성 객체를 정의한 경우,
```
let abc = ref(0);

// .value 를 사용하지 않고 변수 자체에 재할당
abc = 1;

// abc는 ref(0) 인 반응성 객체가 아닌, 원시 값 1을 할당한 변수로 변해버린다. 이렇게 되면, abc 변수는 DOM 을 리렌더링 할 수 없는 변수 상태가 되어버린다.

// const abc = ref(0); 으로 변수를 선언한 경우, abx = 1; 로 원시 값을 할당하면 타입 에러를 발생시켜 바로잡을 수 있다.
```

- const 선언은 변수의 값 자체를 불변으로 만드는 것이 아니라, **'반응성 객체를 가리키는 메모리 주소(참조)를 고정해 Vue의 렌더링 시스템이 고장나지 않도록 코어 레벨에서 보호하는 방식'** 이다.
